Treat this as a shortest-path problem over a filtered graph, not a simple traversal. Using the SQLite C API, load all rows from `backups` and `dependencies`, but keep only edges/nodes where `status = 'SUCCESS'` (excluding failed/in-progress backups is the filtering step naive solutions skip). Build an in-memory adjacency structure from the filtered edges, then run a shortest-path search (Dijkstra, or a topological DP since it's a DAG) minimizing the sum of `size` over the path, considering every SUCCESS 'FULL' backup as a possible start and requiring the path to end exactly at the given target ID via valid parent->child edges. Track back-pointers to reconstruct the ID sequence for the minimum-cost path — don't assume a single path per node, since multiple candidate paths exist and only the cheapest is correct. Write the sequence to restore_plan.txt, one ID per line, from the chosen FULL backup to the target, compile with `-lsqlite3`, run it against ID 99, and sanity-check the total size against at least one alternative path by hand before trusting the output.
