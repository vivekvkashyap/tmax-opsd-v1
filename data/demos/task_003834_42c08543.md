Before touching your own C code, get the vendored library building and passing a basic depth test in isolation. Search the library's Makefile and headers for any macro that caps recursion/traversal depth (grep for terms like `MAX_DEPTH`, `RECURSION`, `HOPS` in the Makefile and source) — a hardcoded `-D` flag in CFLAGS is a classic place for such a perturbation, so raise it well above your required depth (e.g., 100+) and rebuild/reinstall (`make clean && make && make install`, or link to the rebuilt static/shared lib) before writing any application code. Confirm the fix by writing a tiny throwaway test graph deeper than 3 hops and confirming traversal now goes further, rather than assuming the edit worked. For the application itself, implement the alternating-type traversal as a recursive DFS: track the expected next node type (starting at ACCOUNT), only follow an edge if the target's type matches, mark visited node IDs per-path (not globally) to avoid cycles while allowing diamond patterns, and accumulate edge weights only for edges you actually traverse — off-by-one errors in when to check the "next expected type" versus advance it are the most common bug. Compile and link explicitly against the rebuilt library, then test on a hand-built multi-hop CSV before trusting the final integer output.
