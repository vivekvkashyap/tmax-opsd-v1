Start with `start_services.sh` and confirm Postgres/Redis are reachable (psql, redis-cli PING) before writing any C. For the query, don't do it in application code — push the "top 3 per user" logic into SQL with a window function: something like `SELECT followee_id, follower_id, engagement_score, ROW_NUMBER() OVER (PARTITION BY followee_id ORDER BY engagement_score DESC) rn FROM follows JOIN users ON users.id=follows.followee_id WHERE category=$1` wrapped in a subquery filtering `rn <= 3`; this is far cheaper than fetching all rows and sorting in C. In api_server.c: use raw sockets to accept connections, parse the request line for the GET path and the `category` query parameter, then first check Redis with `GET graph:<category>` via hiredis — if hit, write that string straight back as the response body. On a cache miss, run the libpq query, build the JSON manually (nodes sorted by user_id ascending, top_followers sorted by score descending — you already get this order for free from the window-function query if you ORDER BY user_id, rn), `SET graph:<category> <json>` in Redis with no TTL, then return it. Send proper `HTTP/1.1 200 OK` headers including `Content-Type: application/json` and `Content-Length`. Compile with `-lpq -lhiredis`, run it backgrounded, and test both the cold and cached path with curl plus a manual `redis-cli GET graph:tech` to confirm the cache actually got populated correctly.
