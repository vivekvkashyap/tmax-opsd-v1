Write aggregate.c to stream-parse the no-header CSV (fscanf or manual split on comma per line, fields: long timestamp, int sensor_id, double value), accumulating per-sensor count/sum/min/max in a small fixed-size array or hash map keyed by sensor_id (sensor IDs are likely small integers, so a plain array indexed by id, sized generously, is simplest and avoids needing a real hash table). After reading all rows, iterate sensor IDs in ascending order (sort keys if using a map, or just loop 0..MAX if using an array) and print `sensor_id,count,min,max,avg` with `min`/`max`/`avg` formatted via `%.2f`, avg = sum/count. Watch two easy mistakes: sort order must be ascending by sensor_id, and doubles must be printed with exactly 2 decimals (not default precision). For the Makefile, wire `build` to `gcc -O3 aggregate.c -o aggregate`, `run` to execute the binary producing aggregated.csv, and `test` to run the binary 3 times, `md5sum aggregated.csv` after each run, and emit reproducibility_report.txt in the exact 4-line format shown in the task (comparing all three hashes for equality to decide YES/NO). Since the aggregation logic is purely deterministic given the fixed input file, three runs should naturally produce identical output — if they don't, the bug is nondeterministic map iteration order affecting float summation, not randomness.
