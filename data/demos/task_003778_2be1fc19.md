The critical constraint is that the background writer uses `fcntl.flock(LOCK_EX)` around each single-line append, so your rotation script must acquire the SAME exclusive lock (via `fcntl.flock` on a file handle opened on the same path, same lock type) before reading+truncating, and release it immediately after — grabbing the lock only around the copy-and-truncate step (not for the whole script) minimizes the window the writer is blocked, while skipping the lock entirely risks reading mid-write or truncating between the writer's read-modify-write cycle and losing a line. Within the locked critical section: read all current bytes of `data.log`, write them to `data_snapshot.log`, then truncate the original file to zero length (e.g. `f.truncate(0)`), all before releasing the lock. After unlocking, `tar -czf archive.tar.gz data_snapshot.log`, run an integrity check like `tar -tzf archive.tar.gz` (or `tar --test-label`/`-t`), redirecting both stdout and stderr (`> status.txt 2>&1`) to capture the result, then `ln -s archive.tar.gz latest_archive.tar.gz`. Run the script once and confirm the writer process is still alive and continuing to append afterward.
