First, grep `aggregator.py` for the Redis connection port and compare it against the port the actual Redis instance listens on (check `start.sh` or `redis-server` defaults) — a hardcoded mismatched port number is a classic single-line config bug. Second, don't just eyeball the pcap; extract the HTTP payloads with `tshark -r traffic.pcap -Y http -T fields -e http.file_data` (or scapy) to see the literal bytes sent, since edge cases like trailing commas or quoted scientific notation will break a bare `json.loads` — write a tolerant pre-parser (regex-strip trailing commas, coerce numeric strings) before falling back to standard JSON parsing. Third, replace any sum-of-squares variance formula with Welford's online algorithm (track running `count`, `mean`, and `M2`, updating incrementally per sample, with `variance = M2/(count-1)`), since sum-of-squares accumulates catastrophic cancellation error over many updates — this is the only way to hit the 1e-10 MSE bar on 100k requests. Test by starting `start.sh`, posting many values, and comparing `/metrics` against a numpy variance computed independently.
