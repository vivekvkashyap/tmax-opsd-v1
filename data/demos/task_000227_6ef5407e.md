Model "staleness" as a recursive graph problem, not a single flat query. First establish the base staleness set directly from SQL: a table is base-stale if it has no row in `backups`, or if its `last_update` exceeds the `MAX(backup_time)` of its own backups (use a LEFT JOIN + GROUP BY / correlated subquery for the max). Then propagate staleness along `dependencies` transitively: parent stale implies child stale, and this must chase multi-hop chains (A depends on B depends on C), so use a `WITH RECURSIVE` CTE in SQLite that starts from the base-stale table ids and recursively joins through `dependencies.parent_id -> child_id` to pull in downstream tables, unioning results until no new rows are added. In your C program, install libsqlite3-dev via apt if not present, open the DB with `sqlite3_open`, prepare and step through the recursive CTE query with the SQLite C API (`sqlite3_prepare_v2`/`sqlite3_step`/`sqlite3_column_text`), collect the resulting table names, sort them alphabetically (e.g. qsort with strcmp) before printing, and write one name per line to `/home/user/pending_backups.txt`. Compile with `gcc ... -lsqlite3` and run it to produce the file, then verify against the expected staleness propagation by tracing the dependency chain manually.
