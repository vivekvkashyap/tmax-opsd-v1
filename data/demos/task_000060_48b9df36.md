Treat this as a scripted build-migrate-run pipeline where the main pitfalls are linker flags and the dynamic loader path, not the logic itself. In the script: create `/home/user/build/`, compile the C source as a shared object (`gcc -shared -fPIC -o /home/user/build/libcompute.so /home/user/src/libcompute.c`), then compile the C++ program linking against both that library and sqlite3 (`g++ -o /home/user/build/app /home/user/src/main.cpp -L/home/user/build -lcompute -lsqlite3`, with `-Wl,-rpath,/home/user/build` or setting `LD_LIBRARY_PATH` at run time so the loader can find `libcompute.so`, since it isn't installed to a system path). For the database, apply schema then migration in order with `sqlite3 /home/user/build/app.db < /home/user/src/schema_v1.sql` followed by `sqlite3 /home/user/build/app.db < /home/user/src/migration_v2.sql`, creating a fresh db each script run so migrations aren't applied twice. Finally run `LD_LIBRARY_PATH=/home/user/build /home/user/build/app /home/user/build/app.db > /home/user/test_result.log`, and make the script itself executable with `chmod +x`. Test the script standalone end-to-end before declaring done, checking the exact bytes written to the log.
