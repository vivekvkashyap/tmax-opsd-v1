Build the pieces in order and treat the throughput target as a systems-engineering constraint, not an afterthought. First scaffold the filesystem exactly as specified (dirs, `openssl req -x509 -newkey rsa ... -nodes` for the self-signed cert/key, symlinks with `ln -s`, the `target.html` file with exact content, `.bashrc` exports). Then focus on the C server: a naive fork/thread-per-connection or blocking accept loop will not hit 5000 req/s. Use a non-blocking event loop (epoll) or a small thread pool with `SO_REUSEADDR`/`SO_REUSEPORT`, keep the HTTP response pre-built as a static buffer (read `index.html` once at startup, not per-request), disable Nagle's algorithm (`TCP_NODELAY`), and make sure you `close()` sockets promptly to avoid fd exhaustion under load. Compile with `-O2`. Verify locally by starting the server yourself and hammering it briefly with something like `ab` or a quick loop before relying on the provided `/app/bench` tool, since that binary is what the grader will actually invoke — do not try to inspect or reverse it, just make your server generically fast and spec-compliant. Leave the server binary built but not running.
