This task has two coupled deliverables: a strict CLI classifier and a live HTTP-fronted service — get the CLI logic fully correct first since the HTTP path reuses the same validation. In `backup_sanitizer.cpp`, parse each JSON backup graph and implement three independent checks: (1) BFS/shortest-path from every incremental node to any "full" node over `derived_from` edges, rejecting if any such path exceeds 3 hops or doesn't exist; (2) reject if any node's `stored_in` zone has `"status": "compromised"`; (3) sort backups by `timestamp` and reject if any sliding 3-window sum of `bytes` exceeds 500000000000. Also explicitly detect cycles in the `derived_from` graph (e.g. via DFS with a recursion/visiting-state stack) and reject on any cycle — this is easy to miss since a cycle can otherwise pass the hop-count check by never terminating or by being skipped. Test your binary directly against every file in `/app/corpora/clean/` (expect exit 0) and `/app/corpora/evil/` (expect exit 1) before touching the service layer. Then wrap the same validation logic in an HTTP server on port 9000, and edit `/app/nginx/nginx.conf` to add a `location /register { proxy_pass http://127.0.0.1:9000; }` block, restart/reload nginx, and on acceptance push the backup ID to Redis list `valid_backups` (e.g. via `RPUSH`) using a Redis client or raw protocol calls.
