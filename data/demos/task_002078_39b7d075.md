Fix the build before hunting for the numerical bug — a C extension that fails to compile can't be tested at all. Run `python3 setup.py build_ext --inplace` and read the linker error carefully: math functions like `sin`/`cos` used from `<math.h>` require explicitly linking `libm` (`-lm`), which is easy to omit from `setup.py`'s `Extension(...)` args and produces "undefined reference" errors rather than compile errors — add `libraries=['m']` (or equivalent) and rebuild. Once it compiles, don't guess at the bug — write a small delta-debugging harness that calls `/app/math_oracle <x>` as a subprocess for a spread of x values (e.g. sample across the expected input range, including values that stress different code branches) and compares against your extension's output for the same x, printing the divergence per input; bisect the range where errors appear to localize which branch/term of the formula in `fast_math.c` is wrong, then fix just that term. Finally write `run_all.py` to batch-process all 10,000 inputs from `inputs.csv` through your fixed extension and check the output file has exactly 10,000 lines before considering it done.
