Structure the script around the four distinct requirements and don't conflate them. First, filter: for each `.conf` file in staging, read its lines and only proceed if one line is exactly `DEPLOY=true` (not merely containing it). Second, backup-before-overwrite: before touching a live file that already exists, create a hard link (`os.link`, not `shutil.copy`) from the existing live file to `backup/<name>.bak`, removing any pre-existing backup file first so the link call doesn't fail with "file exists" — a hard link preserves the original inode/content even after the live file is later replaced, which is exactly what the verifier checks (same inode/content as the pre-deploy version). Third, atomic deploy: write the new content to a dotfile temp path in the live directory (`.{name}.tmp`), then use `os.replace` (atomic rename, works across the same filesystem) to move it onto the final live filename — never write directly onto the live file in place, and never leave the temp file behind afterward. Fourth, logging: append exactly `Successfully deployed <filename>` per successful deployment, in the order processed. Run the script at the end so the described final state actually exists on disk.
