Work the four sub-problems in order.

1. Linker: run the binary and read the error (`ldd /app/uptime_monitor`). The environment resolves `libuptime_calc.so` to an incompatible version. Find the compatible one under `/home/user/monitor_env/libs`, then point the loader at it correctly (set `LD_LIBRARY_PATH` to that dir, or create the right `soname` symlink). Confirm by re-running.

2. Recover `settings.bin`: treat `disk_image.img` as a filesystem and undelete the file rather than grepping bytes blindly. Inspect with `file`/`fdisk -l`, then use `debugfs` on the image (`ls -d`, `undelete`) or `extundelete` to recover the deleted inode, and write the result to `/home/user/monitor_env/config/settings.bin`. Verify the binary now runs end-to-end.

3. Wrapper caching: since each `fast_wrapper` invocation is a fresh process, persist state on disk. In `fast_wrapper.c`, read a cache file holding the last run's timestamp and captured stdout; if `now - ts < 5s`, print the cached output and exit without exec'ing the slow binary; otherwise run `/app/uptime_monitor`, capture its stdout (e.g. via `popen`), print it, and rewrite the cache. This is what delivers the >=50x burst speedup — the binary sleeps ~100ms per call, so caching must avoid the exec.

4. `regression_test.sh`: assert two consecutive calls within 5s produce identical output and that the second is faster; exit non-zero on failure. Compile to `/home/user/monitor_env/fast_wrapper`.
