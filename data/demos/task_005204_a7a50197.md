Work through the three failure modes independently before wiring them together. First fix the broken symlink: `readlink` the current `active.json` to confirm it points at a nonexistent `development.json`, then `rm` it and recreate it with `ln -s /home/user/netmon/config/available/production.json /home/user/netmon/config/active.json` (use an absolute target so it resolves regardless of cwd). Second, edit `netmon.conf` and flip `autorestart=false` to `autorestart=true` — this is what makes supervisord relaunch the monitor after it exits, which the script does deliberately after each successful ping to simulate a network blip. Third, launch supervisord with `supervisord -c /home/user/netmon/supervisor/supervisord.conf` and confirm it's actually running as a background daemon (check the pidfile it's configured to write, not just that the command returned). Finally, poll `/home/user/netmon/logs/connectivity.log` for a few seconds rather than checking once immediately — the monitor process needs a moment to read the now-valid config and write its success line, and if autorestart isn't set correctly the process will only log once and never again after its first simulated crash.
