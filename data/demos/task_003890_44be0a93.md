Diagnose each layer of the stack independently before touching performance. For the 502: compare the `proxy_pass` target in `/app/nginx/nginx.conf` against the actual socket/address Gunicorn binds to in `/app/api/gunicorn.conf.py` — a mismatched socket path is the likely cause; fix one side to match the other, then confirm with a direct curl to the Gunicorn socket before testing through Nginx. For the VM socket: inspect `/app/start_vm.sh`'s QEMU `-serial unix:...` argument and correct the socket path so it matches what the Flask API expects to connect to, ensure the socket's directory exists, and set permissive-enough permissions (e.g. `chmod` after QEMU creates it, or an `umask`) since you're running unprivileged. Only after `/api/status` returns 200 reliably, address throughput: bump Gunicorn `workers` (roughly CPU-count-based, since one worker serializes all requests) and raise Nginx's `worker_connections`/keepalive settings. Start all three services (Nginx, Gunicorn, QEMU) in the background, then iterate by running `python3 /app/benchmark.py` yourself and reading its parsed `req/s` line before declaring success, tuning further if below 300 req/s.
