Do the heavy lifting in SQL, not C control flow — SQLite's window functions handle both the dedup and the lag-diff in one query. First dedupe: use `ROW_NUMBER() OVER (PARTITION BY sensor_id, epoch_time ORDER BY id DESC)` in a CTE and keep only rows where that rank is 1 (equivalent to "keep the max id per sensor_id+epoch_time group") rather than an aggregate MAX+self-join, which is more error-prone. On top of that deduped CTE, apply `LAG(reading) OVER (PARTITION BY sensor_id ORDER BY epoch_time)` to get `prev_reading`, then compute `reading - prev_reading` as `difference`. Filter out rows where `prev_reading IS NULL` in the outer query (the LAG for the first reading per sensor). Order the final result by `sensor_id, epoch_time`. In the C program, open the DB read-only with `sqlite3_open_v2(..., SQLITE_OPEN_READONLY, ...)` to guarantee you never mutate `backup.db`, run this single query with `sqlite3_prepare_v2`/`sqlite3_step`, and format each row's three floats with `%.1f` when writing CSV lines (plus the header) to `/home/user/report.csv`.
