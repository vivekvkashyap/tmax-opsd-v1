The key design constraint is that the reduction order must not affect the final sum despite chunking — floating point addition is not associative, so summing partial per-chunk totals in different orders (or per-chunk subtotals) can shift the last decimals. Avoid that entirely: regardless of `<num_chunks>`, split the FASTA sequences into files, translate each sequence to its numeric signal string (A=1.1, C=2.2, G=3.3, T=4.4 joined by commas), and call `/home/user/scripts/fft_power.py --signal "..."` once per sequence to collect one power value per sequence — the per-sequence values themselves are always identical no matter how you chunked the input for parallel dispatch, since each sequence is processed independently. Then, before summing, write ALL individual power values (across every chunk) to one flat list, `sort -n` them into a fixed order, and feed that sorted list into `bc -l` with `scale=4` doing a plain left-to-right addition — this fixes the addition order deterministically so 1-chunk and 10-chunk runs always sum in the exact same sequence. Also build the integer-floor histogram from the same per-sequence power values using `awk` or `bc` truncation. Test by running your script with `1` and with `3` or `10` and diffing the `TOTAL_POWER` lines.
