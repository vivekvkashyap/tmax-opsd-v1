The core constraint is single-pass, multi-output streaming — the natural tool is one `awk` program (or a `tee` with process substitution feeding multiple awk/grep branches) that reads events.csv line by line exactly once and maintains state across three concerns simultaneously: (1) a per-region counter of SUCCESS rows seen so far, printing a row to sample.csv only while that region's counter is < 3, still incrementing the counter for every SUCCESS regardless of whether it was printed; (2) an accumulating count of FAILED rows; (3) a running sum of the amount field for every SUCCESS row (not just sampled ones) — these last two do not stop once a region hits 3, they process the entire file. Write header handling explicitly (print header once to sample.csv before processing data rows, and skip the header line when counting/summing). Use awk associative arrays keyed by region for the per-region SUCCESS counters, and write failed_count/success_sum to their own files via awk's `>` redirection or by capturing awk's END-block prints. Wrap this in run_etl.sh, `mkdir -p` the output dir first, `chmod +x` the script, then actually execute it and diff your three output files against manual re-derivation from the CSV to catch off-by-one errors in the per-region cutoff.
