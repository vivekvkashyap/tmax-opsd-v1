Treat the stripped binary as a black-box embedding function you must call, not something to reverse-engineer — the correctness requirement is that the pipeline's own output is internally consistent, not that you replicate the binary's internal math. Build the ETL as a script: parse `products.csv` and `descriptions.json` (join on product_id/id), sort numerically by product_id ascending, and build each combined text exactly in the given `[category] name - desc` format since any formatting deviation changes what embedding the binary produces. Pipe all joined lines into the binary once as a batch (one process invocation, line-delimited stdin) rather than spawning it per-product — that's both required for correctness (consistent embedding space) and for the runtime budget. Parse its raw stdout as fixed 64 float32 little-endian values per line (`struct.unpack('<64f', ...)` or `np.frombuffer(..., dtype='<f4')`), reshape into an (N,64) array, and save with `np.save`. For `recommend.py`, load the saved embeddings once, normalize vectors, and compute cosine similarity via a matrix-vector dot product (`embeddings @ query_vec`) then `argsort` — avoid per-call rebuilding of any index structure so it stays under the latency budget; use FAISS or precomputed normalization if plain numpy isn't fast enough at this scale. Verify by manually checking a couple of queries' neighbors look semantically sane given the input text before considering it done.
