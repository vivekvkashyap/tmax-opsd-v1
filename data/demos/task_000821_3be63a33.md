Fix the build before touching runtime logic. First run `make` in `/home/user/project` to see the actual compiler/linker errors — you'll need `-shared -fPIC` on both the `libencoder.so` and `wrapper.so` targets, and `wrapper.cpp` must wrap the C header include (or the C function declarations) in `extern "C" { ... }` so the C++ compiler doesn't mangle the symbol name `hex_encode`, which is what breaks `-lencoder` resolution. Next, inspect `libencoder.c` for the memory bug: any buffer sized as `len * 2` for a hex-encoded string is one byte short of holding the null terminator — allocate `len * 2 + 1` (or `calloc`) and terminate the string explicitly, since a missing terminator manifests as segfaults or garbage-appended output downstream. Only after `make` builds cleanly and a quick direct call to `wrapper.so` via `ctypes` returns a sane hex string should you write `ws_server.py`: load the shared lib with `ctypes.CDLL`, set `.restype = ctypes.c_char_p` on `encode_string`, run a `websockets` server on `ws://localhost:8765` that echoes the encoded hex, launch it backgrounded, then use a small Python client to send the target string and capture the reply into `/home/user/output.log`.
