Build the validator as four strictly ordered, independently testable checks, since the fuzzer will probe each in isolation. First recover the token by running `tesseract /app/intercepted_upload.png stdout` (try image preprocessing if raw OCR is noisy) and parsing out the string after `TOKEN: ` — compare the `--token` argument to this recovered value with exact string equality before anything else, emitting `ERROR: INVALID_TOKEN`/exit 1 on mismatch. Second, validate the client certificate: parse `--cert` as PEM (Python's `cryptography` library is the robust choice), check it isn't expired (`not_valid_before`/`not_valid_after` against current time) and that its signature verifies against `/app/root_ca.pem`'s public key — don't just check the issuer field matches by name, actually cryptographically verify the chain, since the fuzz corpus includes certs signed by an unknown CA that would fool a naive string check. Third, for path traversal: URL-decode the `--path` value (handling both `%2e%2e%2f`-style and doubly-encoded variants), then resolve it against the `/var/uploads/` base using an approach that catches traversal even when segments would cancel out (e.g. resolve with `os.path.normpath`/`realpath` after joining, then verify the resolved path still starts with the real, normalized base directory) — a naive substring check for `..` is insufficient and will fail many fuzzer cases. Only on all three passing, print `SUCCESS: <absolute_path>` and exit 0. Test manually against a handful of crafted traversal and cert edge cases before trusting broad fuzzer coverage.
