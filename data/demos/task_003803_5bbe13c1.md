This is black-box reverse engineering of a stripped binary — do not try to disassemble it deeply; instead probe its input/output behavior systematically with `strings`, `file`, and by feeding it crafted test files (empty, short, valid-magic, invalid-magic, valid/invalid CRC) and observing stdout and exit codes. Build a mental model of a fixed-size binary header format: check for a "magic" 4-byte signature at offset 0 (try feeding files with different first 4 bytes to bisect what the binary accepts), then look for timestamp/device-id/experiment-id fields and a trailing checksum field, inferring each field's byte-width, offset, and endianness by varying one byte at a time and watching how the printed values change. For the integrity check, compute CRC32 (the standard IEEE polynomial, e.g. via Python's `zlib.crc32` or `crc32` in bash) over the payload bytes following the header and compare against the checksum field to figure out where the payload boundary and checksum field actually live. Once you've derived the exact byte layout and algorithm, reimplement it in Bash using `od`/`dd`/`hexdump` for byte extraction (with correct endianness conversion) and `crc32`/`cksum`-equivalent logic, matching the oracle's exact STDOUT strings and exit codes for every case (TOO_SHORT, BAD_MAGIC, VALID, INVALID). Then build `organize_data.sh` on top of it, using a temp-file-then-atomic-`mv`-rename pattern (same filesystem, so `mv` is atomic) to avoid partial writes, and logging any failures to `corrupt_files.log`.
