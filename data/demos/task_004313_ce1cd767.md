Treat this as a standard gRPC service build: write the .proto exactly to spec (package, service, RPC, message field numbers matter for wire compatibility), then generate stubs with `python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. video.proto` before writing server.py — do this generation step yourself so you can import the generated `_pb2` and `_pb2_grpc` modules directly. In the RPC handler, open the video once with `cv2.VideoCapture`, seek/iterate to the requested inclusive frame range [start_frame, end_frame], and for each frame convert with `cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)`, then compute `np.var(gray)` (population variance, ddof=0 is numpy's default) per frame, and return the arithmetic mean across the range as `average_variance`. A common bug is off-by-one in frame indexing (0-indexed, both ends inclusive) or accidentally computing variance on the BGR frame instead of grayscale — double check both. Serve with `grpc.server(futures.ThreadPoolExecutor())`, add the servicer, bind insecure port on `0.0.0.0:50051`, and call `server.start()` then block (e.g. `server.wait_for_termination()`) in a background process. Your ci_setup.sh should install deps, run protoc compilation, launch server.py with nohup/&, and record its PID; make it executable and actually run it so the service is live for testing.
