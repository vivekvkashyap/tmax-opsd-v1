The key insight is that a leaked process from a previously-tested "bad" commit will make the *next* commit under test falsely appear bad too, corrupting `git bisect`'s results — so your test script must forcibly free port 8080 before it even attempts `make test`, not just rely on the Makefile's own (possibly broken) cleanup. Write a wrapper script that: (1) kills any process currently bound to port 8080 (e.g. via `fuser -k 8080/tcp` or parsing `lsof -i :8080` for a PID to `kill`), (2) runs `make test`, capturing its exit code, (3) kills port 8080 again afterward regardless of the test result so the environment is clean for the next bisect step, and (4) exits with the captured `make test` status so `git bisect run` can interpret pass/fail correctly. Use `git bisect start`, mark `HEAD` bad and `v1.0` good, then `git bisect run ./your_script.sh`. Once bisect converges, capture the identified commit with `git rev-parse <commit>` and write the full 40-character hash to `/home/user/leak_commit.txt`, then `git bisect reset`.
