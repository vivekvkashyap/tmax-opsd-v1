Use `git bisect` methodically rather than manually scanning commits: write a small bisect test script that builds the server at each checked-out commit and feeds it a known malformed-UTF-8 payload over the TCP protocol, marking the commit `bad` if it crashes/corrupts state and `good` otherwise, then run `git bisect start`, mark known good/bad endpoints, and let `git bisect run <script>` converge — this is far more reliable than eyeballing diffs across ~200 commits. Once you've located the bad commit, inspect exactly what changed in `src/parser.c` around UTF-8 validation; the likely defect is a bounds check that got dropped or an off-by-one in the loop that scans multi-byte sequences, causing an over-read on truncated/invalid sequences instead of substituting `?` for the offending bytes. Compare your fixed sanitizer's behavior directly against `/app/telemetry_oracle` on the same malformed inputs (send identical byte sequences to both over TCP and diff the ACK responses) to confirm exact behavioral parity, not just "it doesn't crash." After fixing and rebuilding, start the service so it listens on both the HTTP health port and TCP telemetry port, and write the exact SHA of the offending commit to `bad_commit.txt`.
