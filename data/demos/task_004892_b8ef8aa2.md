Design the Go program around a flexible per-document decode step, since the core challenge is schema polymorphism, not aggregation logic itself. Read each JSONL line into a generic structure (e.g. `map[string]interface{}` or a struct with an `items []struct{Price,Quantity float64}` field plus a separate `TotalAmount *float64` pointer to distinguish "field absent" from "field is zero") so you can detect at runtime which schema variant a record uses; only skip records whose `status` doesn't match the CLI argument. For records with an `items` array, sum `price*quantity` across all items (an empty items array correctly contributes 0). For records with `total_amount`, use that value directly. Accumulate per-user totals in a map keyed by `user_id`. After aggregation, apply the schema validation as a filter step: drop any user whose `user_id` is empty or whose total is not strictly greater than 0 (this naturally excludes zero-total, negative/refunded, and empty-items users). Sort the surviving `(user_id, total_spend)` pairs alphabetically by user_id before marshalling to the required JSON array shape and writing summary.json.
