Since each transaction waits for at most one other, the waits_for relation is a functional graph (each node has out-degree <=1), so cycle detection is simple: build a map tx_id -> (waits_for, wait_time_ms) by parsing the JSONL once, then for each unvisited node follow the waits_for chain recording the path until you either hit a node already fully processed (no new cycle from here) or revisit a node currently on the active path (that revisit marks exactly the cycle — the sub-path from the first occurrence of that node to the end). Deduplicate so each cycle is only reported once (e.g. track globally-visited nodes so you don't re-walk into an already-found cycle from a different start). For each found cycle, sum the wait_time_ms of the edges that lie along the cycle (not edges leading into it) to get total_wait_ms, and find the tx_id with max wait_time_ms among those same cycle-edges for max_wait_tx. Sort the cycle's node list alphabetically for output, and sort the final array by total_wait_ms descending, tie-broken by the first (alphabetically-sorted) element of cycle. Implement this in Rust in src/main.rs using the existing serde/serde_json deps, then `cargo run` to produce deadlock_report.json, and sanity check with `jq` that the JSON structure and key names match exactly what's required.
