First study the log format carefully: each line has a domain in brackets and an event; some lines carry a `CERT_CHAIN_FAIL` marker for a domain, others don't. In `parse_logs.sh`, do a two-pass approach — first scan the whole file to collect the *set* of domains that ever produced a `CERT_CHAIN_FAIL` line (e.g. `grep CERT_CHAIN_FAIL file | grep -oE '\[[^]]+\]' | sort -u`), then in a second pass filter the original file (preserving original order) to keep only lines whose bracketed domain is in that failed-domain set — awk with an associative array built from pass one is a clean way to do this in one script. For redaction, apply a regex substitution (`sed -E 's/[0-9]{4}-[0-9]{4}-[0-9]{4}-[0-9]{4}/XXXX-XXXX-XXXX-XXXX/g'` or equivalent) to every kept line, not just lines mentioning `cc=`. Write the result to `/home/user/output/redacted_failed.log`, creating the output directory first. For `run_sandbox.sh`, construct a `bwrap` invocation with `--ro-bind / /`, `--dev /dev`, `--bind /home/user/output /home/user/output`, and `--unshare-all`, then invoke `parse_logs.sh /home/user/app_logs/server.log` inside it. Run the sandbox script yourself and diff the produced log against your own manual trace of the sample input to confirm exact line order and redaction correctness.
