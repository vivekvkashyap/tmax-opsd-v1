Reverse-engineer the legacy binary empirically before writing any Go code — never assume a format, derive it from observed bytes. Use `xxd`/`hexdump` on small hand-crafted inputs piped through `/app/bin/legacy_tracker` to identify the magic bytes, header field widths/endianness, and per-file record layout (filename length prefix, filename, content length prefix, content) by varying one input at a time and diffing the output tar. Use `tar tvf`/`xxd` on the produced tar to work out the fixed header metadata (mtime, uid, gid, mode, uname, gname) the binary hardcodes into every tar entry — these are usually easy to spot as constant across runs. Feed inputs containing `PASS=...` tokens and `\r\n` sequences specifically to isolate the exact redaction/newline-normalization rules (e.g. does redaction match only up to whitespace? does it replace ALL `\r\n` or just some?) by testing edge cases like `PASS=` at end of content, adjacent `PASS=` tokens, and CRLF at content boundaries.
