Treat the stripped binary as a black box you characterize by observation, not disassembly guesswork alone. Start with `file`, `strings -a`, and `objdump -d` to look for format-string fragments, JSON key names, or timestamp format hints embedded in the rodata section — this narrows the output schema fast. Then do systematic black-box probing: feed the binary single, minimal, controlled input lines (one field varied at a time) via stdin/pipe and diff outputs to isolate each transformation independently — e.g. send lines for one user with several METRIC values in sequence to detect the rolling-window size and whether it uses integer or float division; send different timestamp formats (epoch, ISO-8601, HTTP-date) to see how each is normalized; vary whitespace/casing in ACTION to detect normalization rules. Write a small harness script that pipes many crafted lines through the legacy binary and logs input/output pairs — this becomes your spec. Only after you've pinned down exact rounding (floor vs round), window length, and JSON field order/spacing should you write `new_processor.py`. Finally, validate by piping the SAME batch of test lines through both the legacy binary and your script and diffing byte-for-byte.
