Build this as a small layered service: an HTTP handler, a data-access layer, and a scoring/pagination layer, so each concern is testable in isolation. In the HTTP layer, check the `X-Research-Token` header first and short-circuit with 401 before touching the DB. In the data layer, query SQLite with a parameterized statement (e.g. Python `sqlite3` with `?` placeholders) — never string-format the category into the SQL — so injection payloads in `category` are treated as literal text, not SQL. For each returned id, load and parse `/home/user/data/docs/<id>.json` and merge fields into the exact schema requested. For scoring, shell out to `/app/relevance_scorer` with the merged JSON as a single argv string (watch out for shell quoting/escaping of embedded quotes) and parse its stdout as a float — treat it as a black box, don't guess its formula. Filter by `min_score` (strict `<` excluded), sort by score descending then id ascending for ties, then slice by 1-indexed `page`/`limit` before building the final response envelope. Start the server bound to `127.0.0.1:8080` in the background and verify it responds correctly, including a bad-token and a malicious-category request, before finishing.
