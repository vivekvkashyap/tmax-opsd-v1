Start by connecting to MongoDB with the `mongo`/`mongosh` shell to inspect a sample document from `jobs` and `chunks` so you know the real field names and types before writing any Rust. The performance requirement is the key constraint: do not pull all documents into Rust and filter/join/sort/paginate in application code — that pays serde and network round-trip costs per document and will blow past the time budget. Instead, build a single `db.jobs.aggregate([...])` pipeline that does everything server-side: a `$match` on status/retention_days, a `$lookup` against `chunks` with a `$group`/`$sum` sub-pipeline to total size per job, a `$sort` on completion_time descending, and `$limit` (500) — optionally `$project` to shape the output document directly to the target schema (job_id, total_size_bytes, completion_date, is_archived) so Rust only needs to serialize, not transform. In the Rust code use the async `mongodb` driver with `tokio`, run the pipeline once, stream the cursor, and write JSONL directly — avoid intermediate Vec collection or multiple queries. Compile with `cargo build --release` and time the actual release binary, not `cargo run`, since debug builds are much slower.
