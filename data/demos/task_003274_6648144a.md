Treat this as three separate, testable subproblems inside one script. First, parse `graph.txt` into a dependency map and compute a build order via topological sort (e.g. recursive DFS post-order: build each dependency before the node itself) so that transitive dependencies of the requested target are all resolved before it is built. Second, for patching: before building each component, check for `patches/<name>.patch` and apply it with `patch` — guard against double-application (e.g. track applied patches in a marker file or check if the patch already applies cleanly with `patch --dry-run`) so re-running `make.sh` is idempotent. Third, for the build step itself: concatenate the `.out` files of the component's dependencies in alphabetical order by dependency name, then append the (possibly patched) `src/<name>.txt` contents, ensuring proper newline separation between concatenated pieces — a component with no dependencies just copies its source. Finally write `out/A.out`'s contents to `final_output.log`. Make the script executable, run `./make.sh A`, and manually trace through the dependency graph by hand to confirm the expected concatenation order before trusting the output.
