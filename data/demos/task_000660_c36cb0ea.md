Two performance-critical design decisions matter most here. First, avoid symlink-loop infinite recursion by tracking visited (device, inode) pairs with `lstat`/`stat` as you walk (e.g. `std::set<pair<dev_t,ino_t>>`), and use `lstat` to detect symlinks so you can decide whether to follow them at all — following every symlink blindly is what causes infinite loops, so consider simply not following directory symlinks, only following into real subdirectories, while still verifying regular files (including those reached via file symlinks) once each by inode. Second, do NOT shell out to /app/dataset_verifier per file — instead open each candidate file yourself, parse it as an ELF64 file natively in C++ (validate the ELF magic bytes, walk the section header table using `e_shoff`/`e_shnum`/`e_shentsize`, resolve section names via the section header string table pointed to by `e_shstrndx`), find a section literally named `.research_data`, and check whether its first 4 bytes equal the specific magic constant — implementing this native parse is what gets you under the runtime threshold. For matching files, compute SHA256 (e.g. via OpenSSL's EVP API or a small embedded implementation), copy into /home/user/clean/ renamed by hash, write the manifest CSV, then zip the clean directory and verify the archive opens/lists correctly (e.g. `unzip -t`).
