Two independent halves: an exploit and a fix. For the exploit, read how the Auth Service derives tokens (XOR each byte of the plaintext with the static single-byte key, then base64-encode the result) and write a small Go program that applies the identical transform to the string `role=admin`, printing only the resulting base64 token — implement the XOR+base64 yourself rather than guessing the output. For the fix, rewrite the gateway's auth middleware to (1) stop accepting the old XOR scheme entirely, (2) parse `Authorization: Bearer <payload>.<hex_hmac>`, recompute HMAC-SHA256 over the base64 payload using the given static secret key with Go's `crypto/hmac`/`crypto/sha256`, and compare via constant-time comparison against the hex-decoded HMAC — only accept if it matches and the decoded payload equals the expected role claim. Also add a middleware/handler wrapper that sets the `Content-Security-Policy: default-src 'self'` header on every response, not just protected routes. Finally launch both services as background processes bound to the exact specified host:port pairs and confirm with a quick curl that old tokens are rejected (401) and freshly-computed HMAC tokens are accepted (200) before finishing.
