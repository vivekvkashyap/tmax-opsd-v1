Work phase by phase and test each in isolation before wiring them together. Phase 1: extract the path from the `ACTIVE_CONFIG_PATH = ...` line in the manifest with `grep`/`sed`/`awk`, trimming surrounding whitespace robustly (e.g. `sed -n 's/^\s*ACTIVE_CONFIG_PATH\s*=\s*//p' file | xargs`) so stray spaces around `=` or trailing spaces don't leak into the saved path. Phase 2: the C++ reader must avoid the classic race — open the file, call `fcntl` with `F_SETLKW` and a shared lock (`F_RDLCK`) to block until any writer's exclusive lock is released, only then `mmap` the region and write it to stdout, then `munmap`, unlock (`F_UNLCK`), and close; grabbing the lock is what prevents reading a config file mid-rotation/mid-write by the background writer. Compile with g++. Phase 3: the backup script must read the path file, run the reader binary, pipe its stdout through `gzip` directly into snapshot.gz (`... | gzip > snapshot.gz`, not compress-then-move), `tar -cf` (no `-z`, since gzip is already inside) the gz into snapshot.tar, and `ln -s` a symlink to it. Test the reader against the live-updating file while the background writer runs to confirm no corrupted reads.
