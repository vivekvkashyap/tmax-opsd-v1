Fix the vendored library first, in isolation: run `make` in `/app/libcurvefit-1.0/`, read the actual linker error (it will complain about an undefined math symbol like `exp`), and inspect the Makefile's shared-library link rule — a missing `-lm` flag on the `gcc -shared` command is the classic cause of "undefined reference to `exp`"; add it and re-run `make` until `libcurvefit.so` builds cleanly. Then write the TCP server in C: use a standard `socket`/`bind`/`listen`/`accept` loop on 127.0.0.1:8080, parse the request by reading N then N lines of 100 floats each, allocate a results array of size N, and use `#pragma omp parallel for` over the curve index calling `fit_melting_curve` from the header into per-index slots of that array — because OpenMP workers finish out of order, write into the array by index rather than appending, then after the parallel region iterate 0..N-1 sequentially to emit `Curve <i>: Tm = <value>` lines with `%.2f` formatting, so the client always gets ordered output regardless of computation order. Compile with `gcc -fopenmp server.c -o server -L/app/libcurvefit-1.0 -lcurvefit -I/app/libcurvefit-1.0 -lm`, set `LD_LIBRARY_PATH=/app/libcurvefit-1.0` when launching, and start it backgrounded, confirming with a raw `nc`/socket test that it stays listening after your shell returns.
