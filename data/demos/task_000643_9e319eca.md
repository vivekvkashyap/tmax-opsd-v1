Do not port the legacy tool's per-node query loop into Python — that's exactly the slow pattern you must avoid. Instead push the whole subtree-sum computation into SQLite itself: (1) create an index on `parent_id` (e.g. `CREATE INDEX idx_parent ON backups(parent_id)`) so child lookups are O(log n); (2) write a single recursive CTE that walks from each root down through `parent_id` links, tagging every descendant row with which root it belongs to, then `GROUP BY root_id` and `SUM(size_bytes)` in one query rather than issuing one query per node. Use `sqlite3` module with a single connection/cursor, run the CTE once, and dump results straight to the `roots_summary.json` dict of `{id: cumulative_size}`. Verify correctness first (compare a few subtree sums by hand or with a tiny reference walk) before worrying about speed, then time your script end-to-end — a correct indexed CTE approach should comfortably run in well under a second even at tens of thousands of rows, while a per-row query loop will be orders of magnitude slower.
