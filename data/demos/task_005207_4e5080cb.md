Write `health_check.py` as a simple loop: attempt a `socket.create_connection((host, port), timeout=...)`, close and exit 0 on success, else sleep 1s and retry, tracking elapsed time so it exits 1 past a 15s budget — don't use a fixed retry count, use wall-clock elapsed time so the timeout is accurate. For `start_services.sh`, the core fix is ordering plus PID capture: launch the producer in the background, immediately capture `$!` as its PID, then block on `python3 health_check.py 127.0.0.1 8080` (checking its exit code) before launching the consumer and capturing its `$!` too — write both PIDs to `run.pid`, one per line, only after both processes are actually started. Don't just add a fixed `sleep` before starting the consumer as a "fix" — the whole point of the health check is to remove that race regardless of how long producer startup actually takes. For `log_monitor.sh`, filter with `grep CRITICAL app.log` then extract the 5th whitespace-separated field with `awk '{print $5}'` (check the actual log line format first — logging module output typically prefixes timestamp/level/name before the message, so count columns carefully rather than assuming). Run the full pipeline and let it settle for ~10s before invoking the log monitor.
