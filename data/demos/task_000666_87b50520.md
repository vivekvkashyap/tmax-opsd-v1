In `checksum.c`, the Adler-32-style loop's bug is almost always an off-by-one in the loop bound (`<=` where it should be `<`, or similar) that reads one byte past the buffer — change the comparison so the loop only ever indexes `data[0..len-1]`. For the Go service, use `net/http` for the `POST /process` endpoint, `os/exec` to shell out to the exact `ffmpeg` command given (capture its stdout via `cmd.Output()` or a pipe, and check `cmd.Wait()`/exit status so partial/failed extraction doesn't silently produce a wrong checksum), then use cgo (`import "C"` with a `#cgo` comment pointing at the local checksum.c/.h, passing the byte slice via `C.CBytes` or an unsafe pointer + length) to call `calculate_checksum`, and return `{"checksum": ...}` as JSON. Start the server backgrounded (`&` or a goroutine with the process kept alive) before finishing your turn. Finally cross-compile with `CGO_ENABLED=1 GOOS=linux GOARCH=arm64/amd64 CC=<cross-gcc> go build`, installing `gcc-aarch64-linux-gnu` if needed for the arm64 target, placing binaries exactly at the required names/paths.
