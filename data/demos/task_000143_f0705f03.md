Treat the stripped compiler binary as a black box you probe, not something to disassemble: feed it a handful of your own `.gql` query variants (varying which fields appear in the `MATCH` pattern) through stdin and inspect the SQL it prints on stdout to learn which query shapes translate into `WHERE` clauses filtering on which columns. Your `filter.sh` should: (1) run `/app/graph_compiler < "$1"` to get the SQL text; (2) get SQLite's actual query plan for that SQL against `/app/graph.db` using `sqlite3 /app/graph.db "EXPLAIN QUERY PLAN <sql>;"` (or `EXPLAIN QUERY PLAN` prefixed onto the generated SQL) rather than guessing from the SQL text alone, since query-plan index selection is what actually matters, not surface syntax; (3) grep the plan output for the literal index name `idx_stale_edges` — if present, `exit 1`, otherwise `exit 0`. Never execute the query for real (no SELECT without EXPLAIN QUERY PLAN, no INSERT/UPDATE) so the database stays untouched. Test your script yourself against a few self-constructed clean/evil `.gql` samples covering both index-triggering shapes before trusting it against the hidden corpora.
