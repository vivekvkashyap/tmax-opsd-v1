Since strace/gdb are blocked by anti-debugging, work around it rather than fighting it: the CWE-214 leak means the sensitive key is exposed as a command-line argument of a short-lived child process, so poll `/proc/*/cmdline` (which is null-byte separated) in a tight loop from a script while concurrently launching `./encryptor <dummyfile> <outfile>` in the background, scanning for a new child process whose cmdline contains the target subprocess name and key-like argument. Give the encryptor's fork a brief window (poll every few ms in a loop for a couple seconds) so you don't miss the short-lived process. Once you capture the full argument list, inspect it to identify not just the key but also the encryption tool and its flags (algorithm, KDF, iteration count) — these are your decryption parameters too, since the subprocess is presumably a standard system crypto utility. Reproduce the same command in reverse (decrypt mode) against target.enc using the recovered key and matching flags, and write only the resulting plaintext to flag.txt.
