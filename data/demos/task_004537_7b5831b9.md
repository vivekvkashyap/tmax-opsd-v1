Run `bash -x /app/bisect_anomaly.sh /app/incident_capture.mp4` first and read the trace carefully — the loop variables (LOW/HIGH/MID) will reveal that the update rule never advances the search boundary on one branch, causing it to spin forever. But don't just patch that off-by-one and stop: think about whether binary search is even the right algorithm here — a single corrupted frame surrounded by normal frames isn't a monotonic step function, so bisecting on "is this frame black" can miss or infinite-loop regardless of the off-by-one. Given the video is short (well under a few hundred frames), replace the flawed bisection with a straightforward linear scan: extract each frame's brightness via ffmpeg's `signalstats`/`YAVG` (reuse the existing brightness-check logic) and iterate frames in order until you find the one whose mean brightness drops near zero — that's fast enough here and always correct. Once you have the frame number, stand up a minimal HTTP responder on 127.0.0.1:8888 (a tiny `nc -l` loop or a short Python `http.server`/socket script) that replies `HTTP/1.1 200 OK` with body `{"frame": <N>}` to `GET /anomaly`, run it in the background, and confirm with `curl -i http://127.0.0.1:8888/anomaly` yourself before finishing.
