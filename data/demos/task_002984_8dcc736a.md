Design the archiver around three concerns that each have a specific correct technique — don't conflate them. (1) Concurrency-safe reads: open each file and call `fcntl.flock(fd, fcntl.LOCK_SH)` before reading, retrying/blocking as needed, so you never read a file mid-write by `doc_writer.py`'s exclusive lock. (2) Incrementalism: compare each file's `mtime` (or a stored per-file timestamp) against the value persisted in `/home/user/last_run.txt`, skip unmodified files, and only after a successful backup overwrite that file with the current timestamp. (3) Correct handling per file type: for `.txt`/`.md`, look up `encoding:<filename>` in Redis (`redis-cli` or a Python Redis client) and decode the raw bytes with that encoding before re-encoding to UTF-8 — don't assume UTF-8 going in. For `.media` files, read only the first 32 bytes, hex-encode them, and collect all into a single `media_headers.json` dict written at the archive root — never add the full binary payload to the tarball, since that is what destroys the efficiency score. Build the archive with `tarfile.open(..., mode='w:xz')`. After running, sanity check the efficiency formula yourself (converted text size + header JSON size, divided by the resulting .tar.xz size) to confirm you're above the required threshold before finishing.
