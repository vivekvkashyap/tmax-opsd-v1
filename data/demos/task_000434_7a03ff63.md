Treat this as a Rust CLI that must be both functionally correct on clean data and robust/paranoid on adversarial data, since a downstream oracle binary will crash on bad input. Structure the pipeline in clear passes: (1) parse each row, normalizing timestamp_raw by trying multiple formats (RFC3339/ISO8601 via `chrono`, and fallback to parsing as a raw integer epoch); (2) do adversarial validation FIRST, before any other processing — scan every config_key for shell metacharacters ($, |, >, <) and `../`, and reject (exit 1, no output) as soon as one is found; (3) group rows by config_key, sort chronologically by original (pre-bucketing) timestamp for interpolation and dedup purposes; (4) deduplicate: within a bucket, for identical (config_key, hash) pairs keep only the earliest original-timestamp row; (5) impute missing values via linear interpolation between the nearest earlier and later known values for that config_key (default 0 if none before, carry-forward if none after), rounding to nearest int; (6) after computing final values (including interpolated ones), re-check the numeric range rule (0–10000) since an interpolated value can itself become invalid — this is a common oversight; (7) bucket timestamps to the nearest lower 10s multiple, then sort output by bucket ascending, then config_key alphabetically. Test end-to-end by piping your binary's stdout into the oracle binary and checking its exit code.
