Build sanitize_loc.sh as a line-oriented CSV filter using awk (simplest for this kind of per-row set-comparison logic) rather than trying to hand-roll everything in pure bash. First normalize line endings with `sed 's/\r$//'` or `tr -d '\r'` before any parsing so CRLF input doesn't leave stray `\r` characters embedded in fields. Always emit the header line unchanged. For each data row, extract all `{...}` variable names from Source and from Target separately (e.g. `grep -oP '\{[^}]+\}'`), turn each into a sorted/unique set, and drop the row unless the two sets are exactly equal (same names, regardless of order — don't just compare counts, a swapped variable name with the same count must be rejected). Separately, case-insensitively check Target for `<script>` or `javascript:` substrings (grep -qi) and drop the row if either matches, independent of the variable check. Only rows passing both checks go to the output CSV. For process_all.sh, loop over `*.csv` in the input dir, call sanitize_loc.sh into a temp file, then invoke `/app/i18n_compiler <clean.csv> <output_dir>/<name>.bin`; since the compiler aborts on unbalanced `{`/`}` counts, your sanitizer's variable-set check is what protects it — validate against the given clean/evil corpus examples before trusting your regex edge cases (e.g. `<ScRiPt>` mixed case).
