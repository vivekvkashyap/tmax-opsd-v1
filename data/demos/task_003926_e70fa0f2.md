Write tz_service.cpp so architecture detection uses preprocessor conditionals resolved at compile time (`#ifdef __aarch64__` / `#ifdef __x86_64__`), not runtime detection, since you'll compile the same source twice for two targets. Read TZ with `getenv("TZ")`, defaulting to "NONE" if null. For the locale name, call `std::locale loc(""); loc.name()` — note this reflects the process's LC_ALL/locale environment at runtime, so it must be read at execution time, not baked in. Compile natively for x86_64 with a normal `g++`, and cross-compile for ARM64 with `aarch64-linux-gnu-g++` — verify both binaries' architecture with `file` before scripting the deploy stage. In rolling_deploy.sh, invoke stage 1 directly with `TZ=Asia/Tokyo LC_ALL=ja_JP.UTF-8 /home/user/tz_service_x86 > .../deploy_stage1.log`, and stage 2 through `qemu-aarch64 -L /usr/aarch64-linux-gnu /home/user/tz_service_arm` with the same env-var prefixing pattern but Berlin/de_DE values — the `-L` flag is required so QEMU can resolve the ARM binary's dynamic linker/libs from the cross sysroot, and omitting it is the most common way this fails at runtime with a loader error. Before wiring the full script, manually run each stage's command once and diff its output against the exact expected string format to catch any locale-generation or env-var-passing mistakes early.
