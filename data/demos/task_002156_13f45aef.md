Start the stack with `bash /app/start_services.sh`, then inspect the actual `backup_jobs`/`job_dependencies` schema. The core fix is replacing any Python-side BFS/DFS loop (which issues one query per node — the N+1 pattern) with a single `WITH RECURSIVE` CTE in PostgreSQL that starts from the given job_id, recursively joins `job_dependencies` to traverse all descendants, and collects `(id, job_name, created_at)` for every reachable job. Layer a window function on top: `ROW_NUMBER() OVER (PARTITION BY job_name ORDER BY created_at DESC)` in a subquery/CTE, then filter to `rn = 1` so only the most recent row per job_name survives, and finally `ORDER BY created_at ASC` to produce the restore chain. Execute this as one round-trip using psycopg2 (or similar), extract the ordered `id` list, JSON-encode it, and `SET` it in Redis at `restore_chain:<job_id>`. Profile end-to-end runtime with `/usr/bin/time` to confirm you're comfortably under the 0.5s threshold — an unindexed recursive join or a Python-side sort will likely blow the budget, so lean on the existing `idx_parent` index and let Postgres do the sorting/filtering.
