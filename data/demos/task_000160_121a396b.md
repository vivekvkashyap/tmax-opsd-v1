Treat the stripped `/app/config_diff_oracle` as a black box you must characterize empirically before writing any C code — don't guess its diff semantics. Construct small, targeted pairs of config files (e.g. a key only in A, a key only in B, a key with a changed value, a key containing "secret"/"password"/"token" in each of those three situations) and run the oracle on each pair, recording the exact output line format and how sensitive keys get masked into a separate counter instead of the normal added/removed/modified counts — this reverse-engineering step is what determines your C program's comparison logic, so be thorough with edge cases (case sensitivity of the sensitive-key match, keys present in both with identical values, empty files). Once the logic is nailed down, write the export step (SQLite `content` column dumped to numbered `.txt` files, processed in strict numeric ID order — sort numerically, not lexicographically, since string sort would put "10" before "2") and the C comparator, which should read consecutive file pairs into simple hash maps/sorted arrays of key-value pairs for O(n) diffing rather than any O(n²) approach, since you need a 10x+ speedup over shelling out to the oracle per pair. Wire it all together in `pipeline.sh` with timestamped start/end log lines for each phase, run it, and diff your `summary_stats.log` against oracle output on a held-out sample before trusting the full run.
