Treat the oracle as a black box you can only probe via its input/output behavior — do not try to disassemble or hardcode its binary. First implement the "happy path" parser strictly per the documented record format (magic header, type/timestamp/length/payload/checksum, little-endian, XOR checksum over type+timestamp+length+payload) and confirm it matches the oracle on clean, uncorrupted files you construct yourself. The subtle part is checksum-failure recovery: build small test files with a single byte deliberately corrupted at known offsets, run them through the oracle, and diff the output against several candidate resync strategies (e.g., skip whole record vs. rescan byte-by-byte) to empirically determine how many bytes the oracle advances and from where it resumes scanning for the next valid record header. Once you've inferred the resync rule from oracle behavior, implement identical logic in C++, then stress-test with many corrupted/fuzzed files comparing your program's output byte-for-byte against the oracle's, iterating on edge cases (corruption near EOF, corruption in length field, multiple sequential invalid types) until outputs match exactly.
