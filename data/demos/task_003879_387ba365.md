Start by inspecting the schema (`sqlite3 backup_metadata.db ".schema"`) before writing any query — don't assume column names. For the anomaly detection, use window functions over rows filtered to `status='SUCCESS'`: partition by `server_name`, order by `backup_timestamp DESC`, and use `ROW_NUMBER()` (or `RANK()`) to tag the most recent backup and the 3 most recent per server; then aggregate to get `latest_size` (row_number=1) and `avg_recent_size` (AVG over row_number<=3), filtering with HAVING `latest_size > 1.5 * avg_recent_size`. This typically requires a CTE plus a self-join or a second window-function pass rather than a single flat SELECT. Create the index with columns ordered to match your WHERE/PARTITION/ORDER BY usage (server_name, status, backup_timestamp) so SQLite's query planner can use it — verify with `EXPLAIN QUERY PLAN <query>` and check the output actually says `USING INDEX`, not `SCAN`; if it doesn't, your column order or predicate shape needs adjusting. In the Python script, execute the CREATE INDEX first, then run EXPLAIN QUERY PLAN and dump its raw output to `plan.txt`, then run the real query and write `server_name,latest_size,avg_recent_size` rows to `anomalies.csv` with a header, rounding `avg_recent_size` to 2 decimals.
