Then work top-down through the stack: (1) create the directory for the Python Unix socket with permissions the Gunicorn/nginx user pair can both use (e.g. `mkdir -p` the socket's parent dir, chmod appropriately, and run Gunicorn with `--bind unix:<recovered-socket-path>`); (2) start the Node app with the `PORT` env var set to the recovered port so it binds where nginx expects; (3) edit `/home/user/nginx/nginx.conf`'s two `proxy_pass` lines to point at the recovered unix socket and port respectively, leaving the rest of the config intact; (4) write `deploy.sh` that, in order, ensures directories/permissions exist, backgrounds the Python app (gunicorn), backgrounds the Node app, then starts nginx (`nginx -c <path>` or via the config's own paths), each with `&` or `nohup ... &` so the script itself returns while services persist. After running `deploy.sh`, manually curl both `/python` and `/node` through port 8080 to sanity check before relying on `bench.py`'s 95% threshold — intermittent failures often mean a service wasn't fully up yet or a stale nginx/gunicorn process is still bound to the old (wrong) socket/port.
