Treat this as three independent recovery/analysis steps that feed a final rule, not one big guess. First, recover the deleted .dat files from the ext4 image using filesystem forensic tools (`debugfs -R "lsdel" /app/legacy_data.img` to list deleted inodes, then `debugfs -R "dump <inode> outfile"` to extract each, or use `icat`/`fls` from sleuthkit) — these give you real example files, both clean and crash-triggering, to test against. Second, read the handwritten struct diagram in struct_spec.png with OCR (`tesseract struct_spec.png -`) and manually clean up the recognized text since handwriting OCR is noisy; use it to understand the byte layout (magic bytes, a type/kind byte, and a length field) of the .dat format — don't guess the layout, derive it from the image plus binary inspection. Third, confirm the crash condition empirically: run `ingest.bin` under `gdb` or with `strace`/core dumps on your recovered files and on synthetic variants (vary the byte(s) you believe control the crash) to pin down exactly which field value(s) and thresholds trigger the segfault — use delta debugging (bisect which bytes matter) rather than trusting a single hypothesis. Only once you've empirically confirmed the exact predicate should you encode it in filter.py, which must open each .dat file, parse just the relevant header bytes per your derived spec, and print the basename only when the file does NOT match the crash condition.
