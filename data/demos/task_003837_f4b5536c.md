Treat path validation as the core correctness requirement, not an afterthought bolted onto working I/O. In your Rust restorer, for each manifest entry join the `target` field onto the restore-directory base using ordinary path-string joining, then normalize/clean the resulting path yourself (collapse `.`/`..` components lexically) rather than relying on `std::fs::canonicalize`, which requires the file to already exist and will error or behave unexpectedly on paths you haven't created yet; after normalizing, check that the resulting path's string still starts with the canonical restore-directory prefix (plus a path separator, to avoid a `restored-evil` sibling-dir false-accept) before doing any write. Reject and log the *original* unmodified `target` string (not the normalized one) to `rejected.log`, appended in the same order entries appear in the manifest. For accepted entries, create parent directories as needed, open the file in write mode allowing seeks (create-if-missing, don't truncate on every write since chunks for one file arrive across multiple manifest lines), seek to the given byte offset, and write the chunk's raw bytes read directly from its chunk file. Process manifest lines strictly in file order to match expected log ordering.
