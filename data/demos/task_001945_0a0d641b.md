Build the script as a straight pipeline and get file-locking right since that's the part most naive attempts skip. Parse `DEST_DIR` out of `/home/user/archiver.conf` with something like `grep '^DEST_DIR=' archiver.conf | cut -d= -f2` (don't assume it's always the first line — the config has other keys too), then `mkdir -p` it. Use `find /home/user/drafts -name '*.md' -type f` to enumerate source files recursively across nested subdirectories. For each file, pipe through `python3 /home/user/compress.py < "$file" > "$DEST_DIR/$(basename "${file%.md}").z64"`, then append the manifest line — wrap only the manifest write in a lock: use `flock` on a dedicated lock file (or the manifest file descriptor itself via `( flock -x 200; echo "..." >> manifest.txt ) 200>manifest.txt.lock`) so concurrent invocations can't interleave partial writes; a common mistake is locking around the whole per-file processing loop instead of just the shared-resource write, which defeats the purpose of demonstrating safe concurrent manifest access. Verify correctness by decompressing one `.z64` output back with `base64 -d | python3 -c 'import zlib,sys; print(zlib.decompress(sys.stdin.buffer.read()))'` and comparing to the original file's text, and check the manifest has exactly one correctly formatted line per processed file.
