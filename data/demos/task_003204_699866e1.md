Build this bottom-up: daemon, then supervisor, then the orchestrating rollout script, testing each layer manually before chaining them. Parse `/home/user/mock_fstab` for the line whose filesystem-type column equals `nfs4` and extract its mount-point column (awk with a field check on the fstab's whitespace-separated columns works well) — don't hardcode the path, derive it from the file so the script is robust. `daemon.sh` should be a trivial infinite loop echoing `HEARTBEAT <mount_point>` with a `sleep 0.1` each iteration. `supervisor.sh` needs an outer `while true; do ... ; done` that runs the daemon and redirects its stdout to the stage-specific log — because the daemon should not exit on its own, the auto-restart logic mainly needs to be correct for when it eventually is killed/crashes, so keep the invocation simple and reliable (`"$daemon" "$mount" >> "$log" 2>&1`). In `rollout_monitors.sh`, launch each supervisor with `&` to background it, then actively poll the corresponding log's line count (e.g. a loop checking `wc -l` against 10) rather than a fixed sleep, before moving to the next stage; after both stages clear 10 lines, write the mount point to `success.txt`. Make every script executable with `chmod +x`, and let the background processes keep running when the script exits (don't `wait` on them or kill them).
