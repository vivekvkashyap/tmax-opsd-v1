Design `processor.py` as a producer/consumer pipeline rather than a single loop, since correctness and throughput both matter. Use `multiprocessing` worker processes (or a pool) that each connect to Redis independently and `BLPOP`/`LPOP` from `raw_logs`, parse each entry as JSON, and validate strictly: exactly the 5 required keys present, `id` parses as a UUID and its version is 4 (use Python's `uuid.UUID(id, version=4)` and check `str(u) == id`), and `level` is one of the three allowed values — discard anything else silently. For valid logs, `HINCRBY` (or `HSET` with incremented value) the Redis hash `metrics:counts` at key `f"{service}_{level}"`; use Redis's atomic increment rather than read-modify-write to stay correct under concurrent workers. Separately, run a TCP server thread/process bound to `localhost:9000` that accepts multiple client connections concurrently (track a list of open sockets) and, whenever a worker detects a `CRITICAL` valid log, broadcasts the exact original JSON string plus `\n` to every currently connected client — use a shared queue or Redis pub/sub between the validation workers and the TCP broadcaster since they run in separate processes. Launch `processor.py` in the background and confirm both port 9000 and the metrics via the API are live before finishing.
