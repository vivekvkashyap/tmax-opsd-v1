Work this as a two-source reconciliation problem, not a single script. First, inspect `results.db` for a `-wal` (and `-shm`) sidecar file next to it — a crash can leave committed data only in the WAL, not yet checkpointed into the main DB file. Simply opening the file with a naive reader that ignores WAL will undercount. Connect with the standard `sqlite3` library (Python's `sqlite3` module or the `sqlite3` CLI) — a normal connection automatically replays the WAL on open, so just running a `SELECT SUM(value) FROM records` (or fetching all rows) through a real SQLite connection is enough; do not try to hand-parse the WAL file. Second, for the JSON files remaining in `data/`, don't assume UTF-8: if `json.load` throws a decode error, detect or try alternate encodings (UTF-16LE/BE, UTF-8-with-BOM, etc.) — e.g. read the raw bytes and check for a BOM, or attempt decoding with each candidate encoding until `json.loads` succeeds — then extract `"value"`. Finally, sum the values recovered from the database with the values recovered from the remaining JSON file(s), and write only that integer (optionally with a trailing newline) to `/home/user/final_metric.txt`.
