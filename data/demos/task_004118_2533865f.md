Build this as a real pipeline, then optimize for the 3s budget. First get correctness with `ffmpeg -i /app/data/raw_footage.mp4 -vframes 150 -s 640x360 ...` producing 150 numbered JPEGs, then write the Go program. For concurrency, launch a worker pool (goroutines + a bounded channel or `sync.WaitGroup`) so image reads, gzip compression, and SHA256 hashing overlap across CPU cores instead of running one frame at a time — this is what gets you from the ~4.5-6s sequential baseline under 3s. The trickiest correctness point is the shared audit.log: every goroutine must `syscall.Flock(fd, syscall.LOCK_EX)` the file descriptor, write its full 4-line block, then unlock, or you'll get interleaved/corrupted records that break the "150 well-formed blocks" check. Do NOT serialize compression itself behind the same lock — only hold the lock for the log write, keep compression/hashing outside it. For the manifest, parse audit.log with a proper multi-line block scanner (state machine keyed on "BEGIN FRAME"/"END FRAME") rather than assuming ordering, then compute SHA256 over each written .gz file's bytes. Benchmark with `/usr/bin/time -f "%e"` after building the optimized binary before declaring done.
