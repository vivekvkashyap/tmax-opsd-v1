Diagnose this as a three-layer build/link/runtime chain and fix each layer in order. First, the ABI mismatch: a C header included by a C++ translation unit needs its declarations wrapped in `extern "C" { ... }` (guarded with `#ifdef __cplusplus`) so the C++ compiler doesn't mangle the symbol names differently than the C compiler emitted them — without this you'll see "undefined reference" at link time even though the library exists. Second, the shared-library compile step must produce position-independent code: add `-fPIC` (and `-shared`) when compiling `libdata.c` into `libdata.so`; also make the Makefile create `lib/` and `bin/` (e.g. via `mkdir -p` prerequisites or a phony target) before writing into them. Third, the runtime loader won't find `libdata.so` unless it's on the library search path — either run the binary with `LD_LIBRARY_PATH=lib ./bin/runner` or bake in an rpath at link time (`-Wl,-rpath,'$ORIGIN/../lib'` or similar).
