Treat this as a dependency-ordered startup orchestrator, not a fire-and-forget launcher. For each service: spawn it with subprocess.Popen, then actively poll its target TCP port in a loop (e.g. try socket.create_connection with a short timeout, retry with a small sleep) until it accepts a connection, and only then proceed to start the next dependent process — do not just sleep a fixed amount or start everything at once. Sequence: start db_service, poll port 8001 until open, start api_service, poll port 8002 until open, then spawn the ssh tunnel as a subprocess using the ssh binary with `-i /home/user/.ssh/id_rsa -N -o StrictHostKeyChecking=no -L 9000:127.0.0.1:8002 user@127.0.0.1`, poll port 9000 until it accepts connections, then start monitor_service. Keep references to all Popen objects and wait on them at the end (or loop indefinitely) so the process tree stays alive in the background. Common mistakes: starting processes in parallel without waiting for readiness, forgetting `-N` (which would try to open a shell instead of just forwarding), and not polling the forwarded port before launching the monitor.
