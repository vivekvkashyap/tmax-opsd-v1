Recognize this as a first-match-wins firewall: the oracle walks rules top-to-bottom and stops at the first CIDR match, so reordering changes semantics whenever two rules' CIDR ranges overlap for the same IP pair. Do not simply sort all rules by traffic frequency — first determine, for every (src_ip, dst_ip) pair in `traffic_stats.csv`, which rule the *original* ordering actually matches (its outcome and index), by simulating the same top-down CIDR-match logic in your C program (or by querying `/app/policy_oracle` directly with the base policy file). Then find a reordering that preserves each pair's original matching rule as its *first* matching rule among all rules with overlapping CIDRs, while pulling rules that serve high-frequency pairs as early as possible — e.g. group mutually-non-conflicting rules and sort by aggregate traffic weight, but keep any rule that "shadows" another for a given pair strictly before rules it must continue to shadow.
