This splits into service wiring (Part 1) and a pure algorithmic script that gets fuzz-tested for bit-exact equivalence against a hidden oracle (Part 2) — treat Part 2 as the higher-stakes deliverable and get its logic exactly right. For Part 1, fix `REDIS_HOST`/`REDIS_PORT` in both env files to point at `127.0.0.1:6379`, set the archiver's output path, then start redis-server, the Flask API, and the archiver as background processes and verify end-to-end with a manual test POST before moving on. For Part 2 (`process_metrics.py`): first melt the wide CSV (`timestamp, sensor_A, sensor_B, sensor_C`) into long form, dropping rows with empty values — do this per-sensor-column, not per-row, so a missing value in one sensor doesn't drop the whole row's other sensors. Merge in `location`/`threshold` from metadata by `sensor_id`. Then, critically, sort each sensor's own sub-series by timestamp independently before computing the rolling average — a global sort by timestamp across all sensors mixed together will corrupt each sensor's window. Use a true trailing window of up to 3 (current + up to 2 prior) per sensor, round to 2 decimals, then filter rows where rolling_avg strictly exceeds that sensor's threshold. Finally sort the filtered output by timestamp then sensor_id and print strict JSON with only the specified keys — no extra stdout output, since exact-match fuzzing will fail on stray prints.
