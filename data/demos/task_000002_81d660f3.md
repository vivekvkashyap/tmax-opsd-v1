Build the tunnel first and test it standalone before touching systemd: a TCP proxy accepting connections on `127.0.0.1:9002`, opening a matching outbound connection to `127.0.0.1:9001` for each client, and spawning two threads (or using `select`) to forward bytes in both directions until either side closes — get this working manually (`api.service`'s script running in one terminal, your tunnel in another, `curl localhost:9002`) before wiring in systemd, since debugging unit files is much harder than debugging raw sockets. For the systemd fix, edit `lb.service`'s `[Unit]` section to add both `After=api.service tunnel.service` (ordering) and `Requires=api.service tunnel.service` (so starting `lb.service` pulls the others up too) — `After` alone won't guarantee ordering. Reload with `systemctl --user daemon-reload` then `systemctl --user start lb.service` and confirm all three are active. Only after `curl http://127.0.0.1:9000/` reliably returns the expected body should you write `verify.py`, which just loops 4 times issuing a GET and appending `response.text` (or equivalent) as a line to `/home/user/test_results.log`.
