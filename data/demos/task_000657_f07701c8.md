Treat /app/sensor_oracle strictly as a black box you only ever invoke via its CLI on a file — never disassemble/reverse it, and never hand-derive its formula; let your regression discover the mapping from (features, oracle output) pairs. First install liblapacke-dev (or equivalent) so LAPACKE headers/linking work. Clean raw_sensors.csv with a simple script/awk/python pass that keeps only lines with exactly 5 comma-separated fields that all parse as floats, discarding "N/A"/"ERR"/wrong-column-count rows. Run the oracle once on the cleaned training CSV to produce label values (one float per row, same order). In model.c, build the design matrix with an added bias column of ones (so you solve for 6 weights: 5 feature weights + intercept), and call LAPACKE_dgels (or dgelsd for robustness) to solve the least-squares system in column-major order — a common bug is passing your row-major data directly without transposing, which silently produces garbage weights. Apply the learned weights to test_sensors.csv (also with a bias term) and write one prediction per line to /home/user/predictions.csv. Validate by computing MSE yourself against the oracle's output on a held-out slice of training data before trusting the final threshold.
