Design this as a minimal C-ABI shared library plus a thin ctypes wrapper, and profile before assuming speed is adequate. In C++: implement a WAV reader that parses the RIFF header to locate the `data` chunk and reads 16-bit PCM samples directly (avoid pulling in a heavy audio library — parse the simple WAV format manually or with a small header-only library), a windowed FFT (e.g. Cooley-Tukey radix-2, or link `-lfftw3`/`-lm` if available) applied over frames to compute STFT magnitude/energy, and sum energy across all frames/bins into a single double. Expose it via `extern "C" double analyze_audio_energy(const char* filepath)` so the symbol has a stable, unmangled name callable from ctypes. In `CMakeLists.txt`, set the library target `SHARED`, output name `audio_analyzer` (yielding `libaudio_analyzer.so`), enable optimizations (`-O3`), and link any math library used. In `run_analysis.py`, load with `ctypes.CDLL('./libaudio_analyzer.so')`, set `argtypes`/`restype` explicitly (`c_char_p` in, `c_double` out) to avoid ABI mismatches, time the call with `time.perf_counter()` around just the analysis call (not process startup), and print the result and elapsed milliseconds on separate lines exactly as specified. Iterate on FFT frame size/hop and compiler flags if the 50ms budget isn't met, verifying with repeated runs rather than a single lucky measurement.
