Debug this by fixing one broken layer at a time and testing before moving to the next. Start with the Makefile in `vendored_crc/`: it likely has a bad compiler reference or is missing `-fPIC`/`-shared` flags needed to build a proper `.so` — fix it so `make` produces a valid `libfastcrc.so`, and verify with `nm -D` or a small ctypes test that the exported symbol is callable, without touching `fastcrc.c` itself. Next fix `crc.proto` — look for the specific syntax error (often a missing semicolon, wrong keyword, or malformed message/service block) — then regenerate the Python stubs with `python -m grpc_tools.protoc` (specifying proto_path, python_out, grpc_python_out) so `server.py` can import the generated `_pb2` modules; check `server.py` against the generated message/field names for mismatches. Then fix `gateway.py`'s URL routing/query-param parsing so `/verify` correctly reads `data` and integer `crc` from the query string, calls the gRPC stub, and returns the JSON shape and 400-on-missing-params behavior described. Start both services in the background and curl both the HTTP gateway and gRPC endpoint directly to confirm before finishing — do it in this build-then-test order since gateway failures are often actually gRPC/proto failures underneath.
