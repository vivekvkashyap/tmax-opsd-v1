Separate the two concerns cleanly: a processing script that does the file/config/HTTP work, and `deploy.sh` that only manages the SSH tunnel lifecycle around it. In `deploy.sh`: start the tunnel backgrounded and non-interactively with `ssh -f -N -L 9090:localhost:8000 user@localhost` (the `-f` flag backgrounds it after auth so you don't need a manual `&` plus you still get its PID via `pgrep -f` or by using `ssh ... -L ... &` and capturing `$!` if you skip `-f`), `sleep 2` or so to let it come up, run the processing script, then kill the tunnel process specifically (by saved PID, not a blind `pkill ssh`) so you don't disturb other sessions. In the processing script: for each `<username>.txt` in `/home/user/requests/`, derive `username` from the filename and `role` from the `role: <value>` line, ensure `/home/user/config/site_users.ini` starts with a literal `[Users]` line before appending `username=role`, POST the exact JSON shape with `Content-Type: application/json` to `http://127.0.0.1:9090/sync` (which is now proxied to the real IdP on 8000 via your tunnel), and only `mv` the file into `/home/user/archive/` after a successful response. Make `deploy.sh` executable and run it to confirm the full chain end to end.
