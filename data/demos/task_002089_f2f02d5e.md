Start `metrics_server.sh` in the background (e.g. `nohup ./metrics_server.sh &`) before invoking the worker. Run `strace -tt -o /home/user/pipeline/worker.strace -f ./worker.sh` (use `-f` since worker spawns awk/nc subprocesses) so you capture microsecond timestamps for every syscall, including the crash. Build `timeline.log` by extracting timestamped lines from `metrics.log` plus any strace lines containing "fatal", then merge and sort both sets by their timestamp fields (e.g. with `sort -k1,1` after normalizing timestamp format, or a small awk/python merge) rather than concatenating blindly. For the fix, the root cause is that the naive `sum(x^2) - sum(x)^2/N` formula loses precision on large near-identical floats, producing a negative variance under sqrt; replace it with an online mean/variance update (Welford's algorithm: for each new x, update count, delta = x - mean, mean += delta/count, M2 += delta*(x-mean)) implemented inside a single awk invocation reading data.txt line-by-line, then compute sqrt(M2/count) at END and write the result rounded to 3 decimals to result.txt — avoid re-invoking awk per line as the original script did, since that's also what corrupts precision.
