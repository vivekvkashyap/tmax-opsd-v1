Use a lightweight framework (Flask is simplest) bound to 127.0.0.1:8080 and start it in the background (e.g. `nohup python3 server.py &` or a daemon thread) so it survives after your shell command returns. For `/analyze`, shell out to `ffprobe -v error -count_frames -select_streams v:0 -show_entries stream=nb_read_frames -of csv=p=0 /app/surveillance.mp4` (or an equivalent exact-frame-count method — avoid estimating from duration*fps, which can be off) and append the JSON line under a lock. For every write to `/tmp/video_events.jsonl`, open the file, call `fcntl.flock(fd, fcntl.LOCK_EX)` before writing and unlock/close after — this is what prevents the interleaved/corrupted lines the concurrent test is checking for; a naive open+write without flock will fail under 50 concurrent POSTs. For `/backup`, lock the same file while reading it, convert JSONL rows to CSV with `csv.DictWriter` (headers `timestamp,event`), build the zip with `zipfile.ZipFile`, and reopen it with `zipfile.ZipFile(...).testzip()` to confirm integrity before streaming it back with the `application/zip` content type.
