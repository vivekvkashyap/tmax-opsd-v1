Design the C parser around the record boundaries first: read the log with a simple line-based state machine that resets on `BEGIN_RECORD`/blank-line and captures ID/BASE/FILE fields between `BEGIN_RECORD` and `END_RECORD`, so malformed spacing doesn't break parsing. Implement the Zip Slip check exactly as specified — split the FILE path on `/`, walk the components maintaining an integer depth counter (0 for `.`/empty components, -1 for `..`, +1 for anything else), and flag the record malicious the moment depth goes negative at any point (not just at the end) or if the path starts with `/`; test this logic mentally against a case where a mid-path `..` briefly dips negative then recovers, since order-of-evaluation bugs are the most common mistake here. For the CSV writer, check `access()`/`stat()` for the file's existence before opening (to decide whether to write the header), then open with `open()` using `O_APPEND`, and take an exclusive lock via `flock(fd, LOCK_EX)` (or `fcntl` with `F_SETLKW`) around the write and release it after — don't rely on stdio buffering alone since another process may read between your header check and your write. Compile with `gcc -o detect_zipslip detect_zipslip.c` and run it once to produce the CSV.
