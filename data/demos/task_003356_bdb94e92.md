Two independent fixes in this Go/cgo project. For the linking issue in main.go, the #cgo LDFLAGS currently references the library by name only (-lsolver) without telling the linker where to find it at build time or the resulting binary where to find it at run time; add a library search path flag pointing at /home/user/project/lib and an rpath linker flag (e.g. `-Wl,-rpath,$ORIGIN/../lib` or an absolute path) so the compiled binary resolves libsolver.so without requiring LD_LIBRARY_PATH to be set manually — verify this by running the binary from a clean shell with LD_LIBRARY_PATH unset. For the concurrency bug, inspect hub.go's Hub struct: its map of clients is read and written from multiple goroutines (one per websocket connection) with no synchronization, causing Go's runtime race detector/panic on concurrent map access. Add a sync.RWMutex field to Hub, take a write lock around AddClient/RemoveClient's map mutation and a read (or write) lock around Broadcast's map iteration, releasing with defer. Compile the fixed code to exactly /home/user/project/bin/server (go build -o), then test manually by starting the server and connecting two websocket clients concurrently, sending "solve 5", and confirming both receive "solved: 25" without panics.
