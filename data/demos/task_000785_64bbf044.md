Implement the WAL parser as a strict state machine over raw bytes, and treat RLE encoding as a completely separate second pass. In `wal_parser.c`: open the file in binary mode, verify the 4-byte `DOCW` magic, then loop reading one opcode byte at a time — for `0x01` read a little-endian `uint16_t` length, read that many payload bytes, XOR each with `0xAA`, and append to a growable in-memory buffer (realloc as needed, since text length isn't known upfront); for `0x02` read a little-endian `uint16_t` count and shrink the buffer's logical length by that many characters from the end; for `0xFF` stop reading immediately and discard anything after it, ignoring any bytes beyond that point in the file entirely. Second pass: scan the reconstructed buffer for runs of 4+ identical consecutive characters (capping each run's count byte at 255, starting a new run marker if a run exceeds that), emitting `0x1B <char> <count>` for those runs and literal bytes otherwise, writing the result to the given output path. Then write a bash script that uses `find /home/user/docs_raw -name '*.wal'` to locate inputs, runs your compiled parser on each producing a flattened `.rle` file (basename swap only, dropping subdirectories) into `/home/user/docs_processed/`, then `cd`s there and runs `sha256sum *.rle > manifest.txt` so the manifest lists bare filenames sorted alphabetically (glob expansion is already alphabetical). Verify byte output with `xxd`/`od -An -tx1` on a small test WAL you construct yourself before running against the real directory.
