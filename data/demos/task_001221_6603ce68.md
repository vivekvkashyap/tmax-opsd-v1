Structure this as three separable pieces in `/app/build_service.py`: a DSL parser, a graph algorithm, and a thin Flask/Redis layer. For parsing, walk the input line by line, and when you see `STEP <name>` start a new "current step"; subsequent `REQUIRES <name>` lines attach dependencies to that current step until the next `STEP` line — build an adjacency structure of step -> list of dependencies. For cycle detection and ordering, implement Kahn's algorithm (repeatedly remove nodes with in-degree zero) or DFS-based topological sort with a visiting/visited coloring so back-edges are detected as cycles; if a cycle exists, return HTTP 400 with the exact error JSON before doing any Redis writes. On success, generate a UUID (`uuid.uuid4()`), serialize the ordered step list (e.g. `json.dumps`) and `SET` it in Redis under `build:<uuid>`; on GET, `GET` that key, `json.loads` it and return `{"steps": [...]}`, returning 404 if the key is missing. Watch for two common mistakes: producing a topological order that lists a step before its own dependencies (direction bug — REQUIRES means "comes after"), and forgetting that Flask must run with `debug=False` or in a way that doesn't block your shell (launch with `nohup python3 ... &` or similar) so it stays up for the verifier while still binding to `127.0.0.1:5000`.
