Build this as a clean, generalizable pipeline (it will be re-run against a held-out video, not just the sample), and validate each stage on the sample before chaining them. Stage 1: `ffmpeg -i "$1" -vf fps=<rate> frames/frame_%04d.png` to extract frames at a reasonable sampling rate — dense enough to catch briefly-flashed QR codes without excessive redundant frames. Stage 2: run `zbarimg --raw -q frames/*.png` (or loop per-file) to decode QR payloads, and deduplicate identical consecutive detections since adjacent frames likely capture the same code multiple times. Stage 3: for payloads prefixed `ENC:`, base64-decode the remainder then decrypt with `openssl enc -aes-256-cbc -d -K <hex-of-key> -iv <hex-of-iv> -base64` — note the key/IV strings from the task need to be converted to hex for OpenSSL's `-K`/`-iv` flags (`echo -n "..." | xxd -p`), and pipe the result into `decrypted_keys.txt`, one credential per line, deduped. Stage 4: for URL payloads, parse query parameters (e.g. `redirect_uri`, `next`) and flag as an open redirect any case where that parameter's value is an absolute URL pointing to a domain different from the URL's own host — write only those flagged URLs to `open_redirects.txt`. Parameterize the input path as `$1` so the same script works unmodified against the held-out test video, and aim for high precision/recall since scoring is F1-based across both output files.
