Treat this as a two-layer safety filter, not a single check. First probe `/app/legacy_validator` empirically: run it with a range of crafted UTF-8 strings (ASCII, multi-byte, surrogate-half byte sequences like `\xed\xa0\x80`, very long strings) to learn its exit-code behavior — don't guess its algorithm. Build the safety check as three independent gates on each `text` field: (1) strict UTF-8 validity (decode with `errors='strict'`, catch `UnicodeDecodeError`), (2) reject any code point that is unassigned in the Unicode database (use `unicodedata` — e.g. check `unicodedata.category(ch) == 'Cn'` or rely on the fact that Python strings already reject lone surrogates on strict decode), and (3) actually invoke the binary via subprocess and check its exit code, rather than trying to reimplement its normalization/length logic yourself. A common failure mode is passing the raw bytes to the binary incorrectly (encoding mismatches) or only doing static checks and skipping the live binary call — the corpus is designed to catch entries that only the binary rejects. Parallelize the per-entry checks with `concurrent.futures.ProcessPoolExecutor` or `multiprocessing.Pool`, keep the JSON I/O and CLI-arg parsing in the main process, and preserve input entry order and exact field structure in the output.
