Work top-down: nothing else can be tested until the vendored Docker Compose stack is actually reachable. Open `/app/vendored_usermgmt/docker-compose.yml` and check whether the `api` and `db` services share a common Docker network — if they're on separate `networks:` lists they can't resolve each other's service names at all, so put them on the same network (or remove the split). Then check the `api` service's environment variables against the actual Postgres service name in the compose file — a DB host env var that doesn't match the service's DNS name inside the Docker network will cause connection refusals; fix it to reference the correct service name. Bring the stack up with `docker compose up -d --build`, tail logs, and confirm `curl http://localhost:8000/health` returns 200 before writing anything else. For the health monitor, a simple `while true; do curl -s -o /dev/null -w "%{http_code}" ...; sleep 2; done` loop backgrounded with `&` is enough. For extraction, anchor your grep/awk pattern tightly to the `[USER-MIGRATE] username: ... | email: ... | role: ...` literal format so none of the other 5000 noise lines slip through. For the import script, the 15ms-per-request latency times 5000 sequential requests is the trap — you must fire requests concurrently via `asyncio`+`aiohttp` (a single `ClientSession` with `asyncio.gather` over all POSTs, or bounded concurrency with a `Semaphore` if the API can't handle full concurrency) to land under the 12s wall-clock budget.
