The core difficulty is that the filenames on disk are raw ISO-8859-1 byte sequences that are NOT valid UTF-8, so ordinary C++ string/filesystem APIs (which assume UTF-8 on Linux) will mishandle them — read directory entries as raw bytes (e.g. via `readdir`/`std::filesystem::directory_iterator` treating the path as a byte string, or lower-level POSIX calls) rather than assuming any string is already correctly encoded text. For each entry: keep the raw original bytes for the log's "old path" (the log expects the literal Latin-1 bytes, unconverted, e.g. showing raw 0xE9/0xFC bytes in the log line itself — do not UTF-8-encode the old path in the log), but for the actual rename, transcode each byte >= 0x80 by mapping it directly to the corresponding Unicode code point and UTF-8-encoding it (ISO-8859-1 byte value equals the Unicode code point, so this is a straightforward per-byte expansion, no iconv library needed if you write it by hand). Separately, split the filename at the last '.' to isolate the extension and lowercase only that portion with `std::tolower`, leaving any dot-less filename untouched. Perform the actual `rename()`/`std::filesystem::rename` using the correctly UTF-8-encoded new name, and append one log line per file to `/home/user/normalization.log` in the exact `[old] -> [new]` format before moving to the next file (rename after computing both paths, not before).
