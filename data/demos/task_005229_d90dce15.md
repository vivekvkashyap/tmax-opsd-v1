Use `git bisect` mechanically rather than eyeballing commit messages. In `/home/user/repo`, run `git bisect start`, mark current `HEAD` (or `main`) as `bad` and `v1.0` as `good`, then at each step run `simulate.sh` (or the relevant decode step) and check whether it still converges/decodes correctly, marking `git bisect good`/`git bisect bad` accordingly — automate this with `git bisect run <script>` where the script exits 0 for good and non-zero for bad, since ~200 commits by hand would be slow. Once bisect converges, capture the full 40-char hash with `git rev-parse <bad-commit>` and write it verbatim to `/home/user/bad_commit.txt` (no trailing text, watch for trailing newline expectations). To find the "intended rate," don't trust the corrupted decode path — decode `config.b64` directly and correctly (e.g. `base64 -d` after properly handling any stray characters) and inspect the resulting JSON's `rate` field yourself with `jq`, since the bug likely mangles the base64 payload in a subtle textual way (compare the pre-bug and post-bug versions of whatever line processes the file, e.g. with `git show <commit>^:file` vs `git show <commit>:file`) rather than assuming the number from the buggy output.
