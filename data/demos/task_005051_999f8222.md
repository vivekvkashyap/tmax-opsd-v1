The performance win comes from data parallelism, not from modifying the compiler binary: since each output record maps 1:1 to an input line and order-independence lets you concatenate outputs safely, split INPUT_FILE into N chunks (e.g. `split -n l/<cores>` or `split -l`, using the 4 available cores) into a temp working dir, then invoke `/app/mlist_compiler <chunk> <chunk_out>` on each chunk concurrently — use background jobs with `wait`, or `xargs -P4`/GNU `parallel` if available — and finally `cat` the chunk outputs back together in the original chunk order into routes.db. Watch two correctness pitfalls: chunk outputs must be concatenated in the same order the input was split (don't let parallel job completion order scramble it), and you must byte-for-byte match sequential output size (verify no chunk silently truncated a partial trailing line).
