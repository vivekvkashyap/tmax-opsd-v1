Work this as a three-stage pipeline, then wrap it in a live API.

1. **Video**: Don't rely on ffmpeg's own scene/brightness filters alone — extract frames (e.g. `ffmpeg -i traffic_sensor.mp4 frame_%04d.png`, or read via OpenCV) and compute mean pixel brightness per frame yourself in Python; collect the indices where mean > 200. Frame indexing must match how the video was written (check 0- vs 1-based) so the ping numbers line up with the graph data.

2. **Schema reverse-engineering**: `ls` and inspect each CSV's header/columns rather than guessing from filenames. The nodes file has an id/type/name-style structure covering drivers, vehicles, and pings; the edges file has source/target/relation columns with relations like "drove" and "detected_at". Note that a ping node's *numeric identity* (matching the extracted frame numbers) likely lives in a name/label field, not necessarily its graph id — verify which column actually holds the frame number before matching.

3. **Graph query**: Build the graph with networkx and, for each requested ping id, traverse ping → (detected_at, reversed) → vehicle → (drove, reversed) → driver — it's a two-hop join, not a direct edge. Dedupe and sort results alphabetically.

4. **API**: A Flask/FastAPI service on 127.0.0.1:8080 exposing POST /api/investigate must recompute the traversal from whatever ping list is sent in the request body each call — don't hardcode results from your own test run. Launch it backgrounded/persistent so it survives the request that starts it.
