Start by reading `/home/user/reader.go` closely to learn the exact binary layout it expects (header bytes, per-record field sizes/order, and the 4-byte magic marker value/position it checks) — this tells you the record framing without guessing. In `/home/user/recover.go`, don't assume records are contiguous: scan byte-by-byte (or in a sliding window) for the marker's byte pattern, and whenever a candidate record boundary is found, validate it by checking the marker at the expected offset; if it doesn't match, advance one byte and keep scanning rather than aborting, which is how you skip brownout garbage. For each valid record's value field, read the low 4 bytes and the high 4 bytes separately: if the high 4 bytes equal `0xFFFFFFFF`, the value was written as a padded float32 — interpret only the low 4 bytes as a little-endian IEEE-754 float32 and widen to float64; otherwise interpret all 8 bytes as a standard little-endian float64. Do not naively reinterpret the full 8 bytes as a float64 when padding is present, or you'll get garbage from the exponent/mantissa collision. Write `ID,Value` lines with the value formatted to exactly 4 decimal places.
