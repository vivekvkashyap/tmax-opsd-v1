Treat this as delta-compression against a shared base, not general-purpose compression. First inspect the input directory: sample a few files and diff them to see how much content is shared versus how much varies per file — this tells you whether whole-file compression (gzip/zip) or delta-encoding against a reference will win. Since files are highly similar, pick one file as a reference, store it once (compressed), then for every other file compute a binary patch against the reference using `/app/bsdiff_helper` — run it with `-h` and with no args to learn its exact invocation syntax and argument order before scripting around it, since guessing wrong flags will silently produce garbage patches. Store the archive as a simple custom container: a header/index (e.g. JSON or length-prefixed records) listing each original filename plus its patch's byte offset/length, followed by the reference file bytes and then each patch's bytes concatenated. Write the archive atomically (write to a temp file, then rename) so a crash never leaves a partial file. For extraction, read the index, reconstruct the reference, then apply each stored patch back against the reference using the tool's decode/apply mode to regenerate each original file byte-for-byte, writing outputs atomically as well. Verify correctness locally with a byte-for-byte diff loop before worrying about squeezing size further; only after correctness holds, check the resulting archive size against the target and consider compressing the header/patches (e.g. zlib) if you're close to the limit.
