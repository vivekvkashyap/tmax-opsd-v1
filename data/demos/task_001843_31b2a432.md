Approach this as a standard graph-cycle-detection problem implemented in C with SQLite bindings. First query `locks` to build directed wait-for edges: for each resource, find the transaction(s) with status 'HELD' and the transaction(s) with status 'WAITING' on that same resource_id, then add an edge WAITING_tx -> HELD_tx (a transaction "points to" whichever transaction currently holds the resource it's waiting for). Load all edges into an in-memory adjacency structure (e.g. array of tx_id lists), then run a DFS or simple cycle walk starting from any node to find the single cycle guaranteed to exist — nodes not part of any cycle (transactions that only hold, never wait, or vice versa) should simply be ignored, not included in output. Once you find the cycle as a set of edges, rotate the resulting sequence so it starts at the numerically lowest tx_id while preserving the directed traversal order. Link against `-lsqlite3` when compiling (e.g. `gcc detect_deadlock.c -o detect_deadlock -lsqlite3`), run the binary so it writes `/home/user/workspace/deadlock_report.json`, and validate the output with `jq -c '.cycle'` to confirm exact array formatting.
