Start by getting the build to produce a real shared object: add `-fPIC` to `CFLAGS` and replace the plain `gcc -o libfastcrc.so fastcrc.o` link step with `gcc -shared -o libfastcrc.so fastcrc.o` (also compile the object with `-fPIC`). Then read `fastcrc.c` closely for the performance bug rather than assuming the algorithm itself is necessary: the inner `for j in 0..100` loop only ever XORs `data[i]` shifted by `j % 8`, i.e. it cycles through 8 distinct shift values 12-13 times each — an XOR of the same value an even number of times cancels to 0, so the loop can be collapsed to a closed-form O(1)-per-byte computation (work out algebraically which of the 8 shifted values survive based on parity of repetitions) instead of iterating 100 times per byte. Removing that O(N·100) redundant work is what gets you past the 15x speedup bar — cosmetic tuning like `-O3` alone likely won't be enough. Verify with `/app/benchmark.py` (read-only, don't modify it) and confirm output correctness still matches the checksum semantics the Python reference expects, since speedup with wrong output would still fail an equivalence check if added later.
