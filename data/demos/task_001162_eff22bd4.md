Approach this as build-fix, then trace, then defensive-filter — don't skip straight to writing the filter. First run `python3 setup.py build_ext --inplace` in `/home/user/audioproc` and read the actual error; C-extension build failures are commonly a `sources=[...]` filename in `setup.py` not matching the real `.c` file on disk (check for typos/case mismatches) — fix the reference, not the source file. Once it builds, run the service under `strace -f -e trace=memory,brk,mmap python3 run_service.py /app/trigger.wav` (or use `valgrind --leak-check=full`) and simultaneously read the C extension's parsing code to find which chunk-parsing branch allocates a buffer without a matching `free()`; look specifically at how it walks WAV chunk headers (RIFF/fmt/data and any nonstandard chunk types) since the leak is tied to a specific chunk sequence, not just chunk presence. Use `xxd /app/trigger.wav | less` to inspect the actual chunk layout of the trigger file and confirm your hypothesis matches its structure.
