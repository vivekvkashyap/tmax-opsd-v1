Recover the kernel values from the image first and verify them independently before wiring anything else: run `tesseract /app/kernel.png stdout` (try `--psm 6` for a table layout if default OCR mangles it), then sanity-check the parsed 3x3 grid of numbers makes sense as a filter kernel (e.g. print it) before hardcoding it into your server — OCR mistakes on decimals/negatives are the most likely source of failure here. For calling `libfilter.so` from Python, load it with `ctypes.CDLL`, and set `argtypes`/`restype` explicitly (e.g. `POINTER(c_double)` for arrays, `c_int` for width/height) — the most common bug is forgetting to set `argtypes`, which silently corrupts double-precision args on 64-bit systems. Allocate an output buffer of the right size, pass it and the input as `(c_double * len)(*data)` arrays, and confirm the function fills it in place with a small manual test (e.g. a 3x3 all-ones matrix) whose result you can hand-verify against the known kernel before trusting the full pipeline. Only then wrap it in a Flask/FastAPI `/process` endpoint on 127.0.0.1:8000, run it in the background, and write `test_api.py` to hit it with a known small case.
