Build the pipeline as discrete, testable awk/shell stages rather than one giant one-liner. First, filter: use `awk -F',' 'NF==4'` to silently drop malformed lines — this naturally handles the embedded-newline corruption because a broken record's continuation lines won't have exactly 4 comma-separated fields and get dropped on their own. Watch out for a subtlety: a line that has extra commas inside a field (e.g., 6 fields total) must also be dropped, not just short ones — `NF==4` handles both directions correctly, so don't special-case it. Second, bucket and aggregate in one awk pass: compute `bucket = int(timestamp/3600)*3600`, accumulate sum and count per `(bucket,sensor_id)` key in associative arrays, then in the END block print `bucket,sensor_id,avg` with the average rounded to 2 decimals (use `printf "%.2f", sum/count` — awk's default float printing won't give you clean rounding). Sort the output by bucket then sensor_id to make DB insertion deterministic. Third, load into SQLite: `sqlite3 /home/user/metrics.db` to create the `hourly_aggregates` table, then either pipe a `.mode csv` / `.import` command or generate INSERT statements from the aggregated output — test the import step separately by first dumping the aggregated CSV to a file and inspecting it, since debugging a wrong awk aggregation is much easier before it's inside SQL.
