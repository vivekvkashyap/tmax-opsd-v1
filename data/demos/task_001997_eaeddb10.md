This is a stateful log-replay problem — the graph you need reflects state at the END of the log, not just a static parse of all REQUEST lines, so process events in strict timestamp order and maintain live "holds" (resource -> holding tx) and "waits" (tx -> tx edges) as you go, removing/updating them on RELEASE and ACQUIRE events, not just adding on REQUEST. Concretely: on ACQUIRE, if the resource is free, the tx now holds it; on REQUEST, if the resource is currently held by someone else, add a wait-for edge from the requester to the holder (this edge should reflect final state, so if the holder later releases, that specific wait relationship is stale — think carefully about whether an edge should persist or get reevaluated as holders change); on RELEASE, the tx drops the resource. Load both CSVs, concatenate, sort by timestamp, then replay. Once you have the final directed graph, use networkx's `simple_cycles` for deadlocks and `betweenness_centrality` for the bottleneck.
