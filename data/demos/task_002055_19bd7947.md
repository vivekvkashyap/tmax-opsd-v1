Work in three clean phases and validate each before moving on. First, brute-force the PIN: since the binary exits 1 on a wrong 4-digit PIN, write a bash loop trying every "0000".."9999" value as arg1 with a fixed dummy arg2, and detect success by checking `$?` (or absence of the "Auth failed" message) rather than guessing — this needs at most 10,000 tries and is fast. Second, once unlocked, black-box the encoding: feed the binary controlled inputs (empty string, single repeated characters, strings of length 1/2/3/4/5 to see block-boundary behavior, and the discovered PIN itself as plaintext) and diff outputs to infer the transform. Note the output alphabet/padding pattern is a strong clue it's a well-known encoding applied after some per-byte transform keyed on the PIN characters — confirm by hypothesis-testing (predict output for a new input, run the binary, compare) before trusting your model. Only after you can reproduce the oracle's output for several self-chosen test strings by hand/script should you scaffold the Rust project with `cargo new`, implement the confirmed algorithm using a suitable crate for the encoding step, build with `cargo build --release`, and byte-compare against the original binary (no extra newline) on edge cases like empty input and non-ASCII-adjacent bytes.
