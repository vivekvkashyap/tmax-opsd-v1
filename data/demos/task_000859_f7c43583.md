This is a race-condition capture problem: a file exists on disk for only ~50ms before being unlinked, so a naive "wait for inotify event, then cp" script will lose most files because copying reads the file's content, which takes longer than the window and may race the delete. The robust technique is to react to the filesystem CREATE/MOVED_TO event and immediately create a hard link (`ln`) to the file's inode in your rescue directory — hard-linking is a near-instant metadata operation that keeps the data blocks alive even after the original path is unlinked, unlike `cp` which must read+write the full content. Use `inotifywait -m -r -e create,moved_to --format '%w%f'` on the (possibly not-yet-existing) processing directory, looping with `while [ ! -d ... ]; do sleep`ing until it appears, then spawn the link attempt per event in the background so one slow link doesn't block the next event. After hard-linking, compute `sha256sum` on the safely-linked copy (not the original, which may already be gone) and append to the manifest. Handle the directory not existing at script start.
