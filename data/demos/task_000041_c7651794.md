Diagnose before you fix. Run `systemctl --user status micro-proxy.service` and `journalctl --user -u micro-proxy.service -n 50` to see exactly which step fails; the unit file's `ExecStartPre` reveals a helper script that enforces a disk quota on a proxy-logs directory — inspect that script to learn the size limit and which log directory it checks with `du`, then find what's actually oversized there and clear or truncate it. Separately, trace the `ExecStart` command to find the config file the proxy reads on startup, then follow the directory structure it lives in (typically a "sites-available"/"sites-enabled" pattern) — use `ls -la` to spot a symlink whose target doesn't actually exist (a typo'd filename), and relink it to the real config with `ln -sf`. Write `repair.sh` so every fix is idempotent: use `rm -f`/truncate rather than assuming the log file exists, and always `ln -sf` (which overwrites) rather than `ln -s` (which errors if the link exists). After running the script, `systemctl --user daemon-reload`, start the service, and confirm with `systemctl --user is-active` before redirecting its output to the status file.
