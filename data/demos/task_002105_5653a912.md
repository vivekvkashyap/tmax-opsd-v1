Debug this like a Go concurrency issue, not a rewrite: locate the exact two bugs rather than restructuring the whole handler. (1) In the JSON-decode loop, a `for { dec.Decode(&p) }` that does `continue` on any decode error other than `io.EOF` will spin forever consuming CPU on malformed input — on a non-EOF error you must `break` out of the loop and return an HTTP 400 response instead of looping. (2) The leaked-goroutine bug is a classic unbuffered-channel-plus-select pattern: a worker goroutine sends on an unbuffered channel, but if the request context is cancelled first, the `select`'s `ctx.Done()` branch fires and returns, so no one is left to receive from the channel — the worker goroutine then blocks forever on the send. Fix by making the channel buffered (capacity 1) so the worker can always complete its send and exit even if nobody is listening. After fixing, `go build`/`go run` the service on `127.0.0.1:8080`, and manually test with curl (valid JSON, malformed JSON, and a cancelled request via a short client timeout) while polling `/goroutines` before/after to confirm the count returns to baseline.
