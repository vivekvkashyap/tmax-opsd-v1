Use `git bisect` mechanically rather than manually inspecting commit-by-commit: `cd /home/user/parser_repo`, `git bisect start`, mark current HEAD `git bisect bad`, mark the very first commit (`git log --reverse --format=%H | head -1`) as `git bisect good <hash>`, then automate the check with `git bisect run bash -c 'make clean >/dev/null 2>&1; make >/dev/null 2>&1; ./parser "data file.txt"; test $? -ne 139'` (or check for a nonzero/segfault exit) so bisect runs unattended and lands on the first bad commit — write that 40-char hash to /home/user/bad_commit.txt via `git rev-parse`. Once at the bad commit (`git checkout <hash>`, rebuild), reproduce the crash under gdb (`gdb -batch -ex run -ex bt --args ./parser "data file.txt"`) or via `ulimit -c unlimited` + core dump analysis (`gdb ./parser core -ex bt`) to get the backtrace; the top frame gives the crashing function name for crash_function.txt. Cross-reference the backtrace's line number against main.c *as it exists at that bad commit* (not HEAD) for crash_line.txt — checking out the wrong revision when reading the file is the easy mistake here.
