This is a systemd unit-ordering problem, not a scripting problem. Edit `/home/user/.config/systemd/user/vm-launcher.service`'s `[Unit]` section to add both `Requires=prepare-disk.service` (so starting vm-launcher pulls in and requires success of prepare-disk) and `After=prepare-disk.service` (so it actually waits until prepare-disk completes before running) — `Requires=` alone only affects activation, not ordering, so both directives are needed together. Leave the `[Service]` section untouched. After editing, run `systemctl --user daemon-reload` (making sure `XDG_RUNTIME_DIR` is set correctly for the user's systemd instance) so the unit changes are picked up, then `systemctl --user start vm-launcher.service`. Confirm success by checking `journalctl --user -u vm-launcher.service` or `systemctl --user status` for errors, and finally verify `/home/user/vm_run.log` contains the success line the Python script writes once it finds the disk symlink in place.
