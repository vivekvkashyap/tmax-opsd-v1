Treat `/app/provision_worker` as a black box and attack redundant recomputation, not the binary itself: its output for a repo only changes when that repo's commit changes, so cache stdout keyed by current HEAD commit hash instead of trying to speed up the binary. First probe empirically — run it a few times against a test repo, time it, confirm output is stable per-commit. Design a C++ TCP server on 127.0.0.1:8080 that, per request, gets the repo's current HEAD (cheap `git rev-parse HEAD`, not the slow worker) and serves a cached result if present, else invokes the worker once, caches by hash, and serves it; use a thread pool so concurrent requests sharing a cached commit don't block on redundant invocations. Wire the bare repo's `post-receive` hook to refresh/invalidate the cache on push. Benchmark against 100 sequential baseline calls to confirm >=3x speedup.
