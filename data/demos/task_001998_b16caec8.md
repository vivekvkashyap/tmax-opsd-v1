For the vendored compiler, locate the file doing the directory walk (likely `bin/parser.sh`) and inspect its `find` invocation: a `-L` flag makes `find` dereference symlinks and will spin forever on a cycle — remove `-L` (or replace with `find` default, which does not follow symlinks for traversal) so `.md` files are still found via normal directory entries. For the loop detector, do not naively `readlink -f`/`realpath` on the whole tree (that can itself follow a cycle) — instead walk the tree yourself (e.g. `find -type l` plus manual traversal, or a recursive function) tracking the chain of *visited inode/device pairs* (from `stat -c '%d:%i'`) for the current descent path; if you re-encounter an already-visited inode within the same descent chain, that's a loop, exit 1. Clear the visited set per top-level branch so siblings don't false-positive. Test against both a deep valid tree and deliberately constructed loops (`ln -s . x`, and mutually pointing dirs) before trusting it. For the manifest step, parse the log's multi-line `PATH:` records, run your validator on each unique path, keep only exit-0 ones, then `find <dir> -name '*.md' -exec sha256sum {} +`, and sort the combined output by path before writing.
