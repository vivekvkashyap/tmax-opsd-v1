This is a two-part git forensics task: recover a secret from history, and bisect a performance regression. For the secret, walk the repo's history rather than just the current tree — `git log --all -p -- '*.sh' '*.conf' '*.cfg' '*.env'` or `git log -p | grep -i api_key` across all commits/branches to find where a config file was accidentally committed with a hardcoded key before being scrubbed in a later commit; note it may only exist in an intermediate commit, not HEAD. For the leak, write a small test wrapper that runs `monitor.sh` (with the recovered `API_KEY` set) for a fixed short duration and measures RSS growth (e.g. via `ps` or `/proc/<pid>/status` sampled over time) to confirm memory grows unboundedly. Then use `git bisect start`, mark a known-good early commit and the current HEAD as bad, and let bisect walk through commits running your memory-growth test script at each step (`git bisect run <script>`) to programmatically pinpoint the exact commit where growth begins — this is faster and more reliable than manual inspection. Once found, get the full 40-character hash via `git rev-parse <commit>` and write both values to `/home/user/report.txt` in the exact `API_KEY=...` / `BAD_COMMIT=...` format.
