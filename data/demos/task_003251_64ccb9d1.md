Work in order: first generate the traffic, then write the program, then compile, then run with the right environment — doing these out of order (e.g. compiling before traffic exists) risks a zero-byte counter passing accidentally but failing the "strictly greater than 0" check if you generate traffic after reading stats. Start `python3 -m http.server 8080 &`, confirm it's listening (e.g. a quick `curl -s -o /dev/null -w '%{http_code}'` check), then issue the real `curl http://127.0.0.1:8080` before your monitor program runs — the rx/tx counters must reflect traffic that already happened by the time the C++ program reads them. In the C++ source, read `/sys/class/net/lo/statistics/rx_bytes` and `tx_bytes` with plain ifstream reads (they are single-line numeric files), and get local time via `std::time`/`localtime` (which respects `TZ`) rather than hardcoding a timezone offset — never bake in a fixed UTC offset since `TZ` must actually control the output. Format the timestamp with `strftime` into exactly `[YYYY-MM-DD HH:MM:SS] RX: <n> TX: <n>`, matching the regex precisely (single space after each colon, no trailing newline issues). Compile with `g++ -o /home/user/net_monitor /home/user/net_monitor.cpp`, then invoke it as `TZ=Asia/Tokyo /home/user/net_monitor > /home/user/network_log.txt` so the environment variable is set specifically for that execution, not just exported globally beforehand.
