Parse the JSONL into a directed graph mapping job_id -> depends_on list. To find cycles/deadlocks, run Tarjan's or Kosaraju's strongly-connected-components algorithm (or repeatedly DFS with a recursion-stack visited set) over the dependency graph — any SCC with more than one node, or a self-loop, is a deadlock; collect its job_ids, sort each cycle's ids alphabetically, then sort the list of cycles by first element. Next, build the "stuck" set: any node reachable from (i.e. transitively depending on) a deadlocked node is also unschedulable — do a reverse reachability walk (or just repeatedly check "any of my direct/indirect deps is in a deadlock or already stuck") starting from the deadlocked node set, and exclude all of those plus the deadlocked nodes themselves from tiering. For the remaining nodes, compute each job's tier as 1 + max(tier of its dependencies), with tier 0 for jobs with no dependencies — process nodes in dependency order (e.g. via a queue/BFS on the DAG, or memoized recursion) rather than naive iteration, since a job's tier depends on all its deps already being resolved. A common mistake is computing tiers via simple depth without accounting for jobs having multiple dependencies at different depths — always take the max, and remember to alphabetically sort ids within each tier list in the final JSON.
