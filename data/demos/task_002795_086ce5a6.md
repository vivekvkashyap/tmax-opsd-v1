The nondeterminism comes from concurrent `#pragma omp atomic` additions happening in unpredictable order, which changes floating-point rounding run to run — fix it by giving each OpenMP thread its own accumulator slot (e.g. a fixed-size `double partial[omp_get_max_threads()]` array indexed by `omp_get_thread_num()`, each thread only ever adding to its own slot inside the parallel region), then sum that array sequentially in a fixed index order *after* the parallel region ends — this guarantees the same operation order every run regardless of scheduling. (A `reduction(+:gradient)` clause is not guaranteed identical across compilers/runs the same way an explicit fixed-order sum is, so prefer the manual approach if you want a stronger determinism guarantee.)
