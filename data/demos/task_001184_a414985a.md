Separate the two subsystems: video decoding to recover edges.txt, and a from-scratch C graph tool that will be fuzz-tested against a hidden oracle, so implement the spec precisely rather than approximating. To decode the video, use ffmpeg to scale each frame down to 2x1 pixels in grayscale (e.g. `ffmpeg -i /app/dataset_video.mp4 -vf scale=2:1 -pix_fmt gray -f rawvideo -`) and read the raw byte stream two bytes at a time per frame — since the encode is lossless (crf 0 / rgb-space h264 or similar), the left pixel byte is the source ID and the right pixel byte is the target ID; write one `Source Target` line per frame in decode order to edges.txt (dedup only if that matches what you observe, but don't assume — verify against a few frames manually first, e.g. with ffprobe/hexdump). For query.c: build an adjacency list from edges.txt, sort each node's neighbor list ascending so BFS naturally explores lower-ID neighbors first, run standard BFS from U tracking parent pointers, and when a node is reached via multiple equal-length paths, keep the parent that yields the lexicographically smaller path (compare candidate predecessor node IDs, not just recency) — reconstruct and print the path in the exact "Path: U->X->Y->V" format, or "NONE" if unreachable, looping over stdin queries until EOF.
