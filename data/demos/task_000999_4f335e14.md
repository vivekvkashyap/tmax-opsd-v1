Treat this as a performance-critical systems program, not a simple file mover — a naive single-threaded design will not hit the 3-second budget for 50,000 files. Start services first and confirm Redis/Flask respond before writing organizer.cpp. Fetch /rules once at startup via libcurl to get target_dir and prefix (don't re-fetch per file). For traversal, walk /app/raw_data recursively (e.g. `nftw`/`std::filesystem::recursive_directory_iterator`) collecting all .dat paths, then process them in parallel using a thread pool (std::thread with a work queue, sized to hardware_concurrency) since single-threaded I/O plus locking for 50k files will likely blow the time budget. Per file: open, `flock(fd, LOCK_SH)` (or exclusive) before reading to avoid partial reads from the still-writing simulator, read the 4-byte little-endian sensor_id then 100 floats, compute the mean, then rename/move the file to `target_dir/{prefix}_{sensor_id}.dat` (rename is atomic and fast — prefer it over copy+delete). For the shared summary.csv, open once, and guard each append with `flock` on that file descriptor (hold the lock only for the write, not the whole computation) to avoid interleaved writes across threads. Batch Redis SADD calls (pipelining or a single connection per thread, not one connect per file) to avoid per-file connection overhead dominating the runtime. Compile with `-O3 -pthread` and link `-lcurl -lhiredis` (or use raw RESP protocol over a socket if hiredis is unavailable), then run /app/verify_performance.sh yourself to measure before declaring done.
