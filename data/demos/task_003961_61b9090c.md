First recover the two hidden compliance parameters by running OCR (e.g. `tesseract /app/compliance_rules.png -`) on the image and parsing the `MIN_AMOUNT=` and `MAX_HOPS=` lines from the output text — don't guess or hardcode values, derive them programmatically or read the OCR text yourself. Then design the C++ program: read T transactions, build an adjacency list only from edges whose amount is >= the recovered MIN_AMOUNT (dedupe parallel edges between the same source/destination pair), then for each query run an unweighted BFS from source, capping expansion at the recovered MAX_HOPS hops, printing the hop count or -1 if destination isn't reached within that bound. Use fast I/O (scanf/cin with sync_disable, or manual buffered reading) since the fuzzer streams thousands of large randomized cases and stdout must match the oracle exactly on every one — off-by-one hop-limit handling (whether MAX_HOPS is inclusive) is the most common place naive solutions diverge from the oracle, so verify carefully with small hand-built graphs before compiling with `g++ -O3`.
