The bottleneck is per-record invocation overhead of `/app/anonymizer`, so the whole design must avoid calling it once per user_id. Stream-process the 50 CSVs (e.g. with pandas in chunks, or Polars/DuckDB) filtering to the two relevant event types, then collect the *unique* set of user_ids across the whole dataset and pipe them to the anonymizer binary as one batched newline-delimited stdin write, reading back the corresponding output lines in order to build an in-memory id-to-anonymized-id map (or split the unique-id list across several parallel anonymizer subprocesses if that's faster) — never shell out to the binary per row. Once you have the mapping, do a vectorized join/replace of user_id, extract the calendar date from the timestamp, and group-by (date, anon_user_id) summing duration_ms. Write the result with exactly three named columns to Parquet (e.g. via pandas `.to_parquet` or pyarrow) at the required path. Wrap the whole pipeline in the specified shell/python entry script, and time your own run locally against the runtime threshold before considering it done — correctness (exact grouped sums) matters as much as speed.
