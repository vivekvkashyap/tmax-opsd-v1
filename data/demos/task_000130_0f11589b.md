This is a multi-service system with a security-critical Rust component — get each piece working in isolation before wiring start_all.sh. In main.rs: use a Redis client crate (e.g. redis-rs) to BLPOP the dataset_queue list in a loop; for each popped tar path, before extracting anything, validate every entry's header (using the `tar` crate or manual header parsing/mmap) — reject the whole archive if any entry path is absolute, contains a `../` component, or is a symlink whose target (resolved relative to the extraction root) would escape that root; do this validation pass fully before any file is written to disk, since a partially-extracted evil archive would still fail the test. Only on a fully-clean archive, extract to extracted/<name>/ (name = tar filename minus `.tar`), then walk the extracted files, sha256-hash each, and write the manifest as `<hash>  <relpath>\n` lines sorted or in walk order as required. For hardlinking identical files across datasets, maintain a hash->existing-path map, and wrap the actual `link()` syscall (or `std::fs::hard_link`) critical section with `flock` on the shared lock file so concurrent processing can't race on the same target link. In start_all.sh, launch redis-server, the existing Python uploader (uvicorn/python app.py), and your built Rust binary, each backgrounded, then wait/poll until ports 8000 and 6379 respond before exiting the script.
