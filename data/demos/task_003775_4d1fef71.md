Parse `/home/user/logs/config_updates.log` with a simple state machine that looks for `[BEGIN_UPDATE]`...`[END_UPDATE]` blocks, extracting `TARGET_FILE` and `STATE`, and only keeps entries where `STATE: COMMITTED`. For the binary header, use precise fixed-width writes: 4 raw ASCII bytes `BCKP`, then a 2-byte little-endian unsigned short for version=1 (`uint16_t` written low-byte-first), then a 4-byte little-endian unsigned int for the entry count — verify byte order explicitly rather than relying on native struct layout, since host endianness/padding can silently break the exact-byte-header requirement. For the payload, store the first (base) config file's contents in full, then for every subsequent version compute a diff against the immediately preceding version of the same target file (e.g., invoke `diff` via `popen` and capture stdout, or implement your own line-diff) and store only that delta plus enough metadata to identify which base/previous version it applies to. Compile with g++, run the tool, and check `stat -c %s /home/user/archive.bin` is comfortably under the 250,000-byte threshold — if not, your diffs aren't small enough and you likely stored full files somewhere.
