Reverse-engineer the binary format purely from the written spec, not by disassembling `/app/doc_encoder` — treat the legacy binary only as a black-box oracle you can feed test inputs to and diff outputs against while iterating. Build the encoder in stages and test each independently: (1) run-length-encode the raw bytes into (count,byte) pairs, splitting any run longer than 255 into multiple blocks of at most 255; (2) compute checksum = sum(all input bytes) % 256; (3) XOR every byte of the RLE payload (both count and byte fields) with that single checksum value; (4) prepend the 4-byte ASCII "DOCX" magic, then the original length as a 4-byte little-endian unsigned int (use struct.pack('<I', ...)), then the 1-byte checksum, then the XOR'd payload. Pay special attention to the empty-input edge case and to runs that are exactly 255 or 256 bytes long (256 must become two blocks: 255+1). Validate by generating varied test inputs (empty, single-repeated-byte, mixed, near-boundary run lengths) yourself, running them through `/app/doc_encoder` and your Python script, and diffing the raw stdout bytes with `cmp` or `diff` before considering it correct — don't just eyeball hex.
