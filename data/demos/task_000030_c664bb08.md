Build and verify each layer independently before wiring them together, since failures here compound. Write `telemetry.proto` first and generate C++ stubs via `protoc --grpc_out --cpp_out` to confirm it compiles cleanly against the installed grpc++/protobuf versions. In `bridge.cpp`, run the gRPC client stream-reading loop and the websocketpp server's `run()` loop on separate threads (e.g. std::thread) so neither blocks the other — a common naive mistake is running both synchronously on one thread, which stalls broadcasting. On each `Metric` received, build the JSON string manually or with nlohmann::json and call the websocketpp server's broadcast/send to all open connection handles (track them in a set added/removed via open/close handlers). For `CMakeLists.txt`, explicitly find_package for gRPC, Protobuf, Threads, and Boost, and link them all to the `telemetry_bridge` target. For `build_and_package.sh`, run the actual cmake+make build, mkdir the dist/bin and dist/lib tree, copy the built binary, then run `ldd path/to/telemetry_bridge`, parse out the "=> /path" resolved library paths (skip lines without such a path, e.g. linux-vdso.so and the dynamic linker itself if desired), and cp each into dist/lib. Test the packaged binary actually launches with `LD_LIBRARY_PATH=dist/lib` before considering it done.
