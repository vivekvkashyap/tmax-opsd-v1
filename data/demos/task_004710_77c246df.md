Split this into a C build fix and a Python FFI/websocket wrapper. For the Makefile, add a target that compiles `algo.c` to a shared object: `gcc -fPIC -c algo.c -o algo.o` then `gcc -shared -o libalgo.so algo.o` (or a single `-shared -fPIC` invocation), making sure `all`/default target actually produces `libalgo.so` (not just the old standalone `algo` binary) and that `compute_root` remains a plain exported C symbol (no `static`). For `ws_server.py`, use `asyncio` + the `websockets` library's `serve()` bound to `localhost:8765`; inside the connection handler, load the library once with `ctypes.CDLL('./libalgo.so')` (or an absolute path built from `__file__`), and explicitly set `lib.compute_root.argtypes = [ctypes.c_double]` and `.restype = ctypes.c_double` — omitting this is the usual bug since ctypes defaults to interpreting return values as `int`, truncating your double results. On each received text message, `float()`-parse it, call the C function, and send back `str(result)`. Before entering the serve-forever loop (but after the socket is actually listening), write the literal string `SERVER READY` to `server_ready.log`. Run `make` to build the .so, but don't block the terminal running the server yourself — just leave the script ready for the harness to launch.
