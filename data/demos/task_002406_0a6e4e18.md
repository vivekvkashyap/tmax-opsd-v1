Structure this as: per-file extraction functions dispatched in parallel, then a serial merge/sort/write step (parallelism must not scramble ordering of the final output). For each file, branch on extension: `.json` -> `jq -r '.app.port, .app.memory'`; `.xml` -> `xmlstarlet sel` or a targeted `grep -oP`/`sed` on `<port>`/`<memory>`; `.ini` -> parse the `[app]` section with `awk`/`grep` for `port=`/`memory=`. Normalize memory in a small shell function: strip trailing `M` and use as-is, or strip trailing `G` and multiply by 1024 (`$(( ${val} * 1024 ))`). To parallelize, have each worker (background `&` job or `xargs -P N -I{}` calling a helper function/script per file) write one CSV row to a temp file or emit to a shared file with locking, then after `wait`, sort all data rows by filename (`sort -t, -k1,1`) and prepend the header. Write the executable script, run it, and diff-check your CSV structurally against a hand-built example before trusting the parallel path at scale.
