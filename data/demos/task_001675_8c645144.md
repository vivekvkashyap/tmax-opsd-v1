Model this as graph invalidation + topological build ordering, which is the standard incremental-build technique (like `make`/Bazel). Steps: (1) Recursively find every `build.json` under `/home/user/polybuild`, parsing `name`, `deps`, `cmd` into a dict keyed by target name — build both a dependency graph and a reverse (dependents) map. (2) Load `/home/user/polybuild/cache.json` (name -> last cmd). (3) Compute "dirty" via propagation, not a single pass: a target is dirty if it's missing from cache, if its current cmd differs from the cached cmd, OR if any of its dependencies (transitively) are dirty — do this with a fixed-point/BFS from the initially-dirty set outward through the dependents map (or memoized DFS) so cascading is captured, since checking each target only against its own immediate cmd/cache miss is the common mistake that misses cascade dirtying. (4) Topologically sort ALL targets (e.g. Kahn's algorithm using `deps`), then filter the sorted order down to just the dirty ones and emit their `cmd`s, one per line, after a `#!/bin/bash` first line, into `/home/user/build_plan.sh`. (5) Write name->cmd for every target (dirty or not) as `/home/user/new_cache.json` via `json.dump`. Execute the script and inspect both output files afterward.
