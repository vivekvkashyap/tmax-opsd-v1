Approach this as delta debugging against a black-box binary. First run `/app/telemetry_ingest` against each file in `/home/user/raw_logs/` individually to find which ones crash, then bisect each crashing file line-by-line (binary search: feed halves, then quarters, etc.) until you isolate the single minimal JSON line that reproduces the panic. Once you have one or two minimal crashers, vary their fields one at a time (change a key's value, remove a key, change its type) to determine exactly which key/value combination triggers the `unwrap()` panic — don't stop at the first crashing example, confirm the general rule by testing near-miss variants (e.g. same field present but non-null, or same value but a different surrounding field) so your filter isn't overly broad or narrow. Once the rule is nailed down, write `/home/user/sanitize.sh` as a thin `jq` (or equivalent) filter over stdin that passes through lines not matching the dangerous condition unchanged (byte-for-byte, no reformatting) and drops the rest, then `chmod +x` it. Validate by piping your isolated crash lines and clean lines through the script and confirming stdout matches expectations before assuming it generalizes.
