Two independent defects must both be fixed: precision and reduction order. First, change every relevant numeric type in `compute_spectrum.cpp` from `float` to `double` (sequence values, angles, sin/cos results, real/imag accumulators, and the output array) — mixing `float` math against a `double`-precision reference will never match bit-for-bit or within tight tolerance. Second, eliminate the OpenMP nondeterminism: `#pragma omp atomic` accumulation into a shared array from a parallel loop means different threads' contributions land in a different order on every run, and floating-point addition is not associative, so results vary run to run. Restructure so each thread/iteration writes its per-sequence power spectrum into its own private/indexed buffer (e.g. a 2D array indexed by sequence i), then after the parallel region do a single-threaded sequential loop that sums buffer[i] across all i in a fixed, deterministic order — this guarantees the same floating-point summation order regardless of thread count or scheduling. Recompile with `-fopenmp`, run once to compare against `reference_spectrum.txt`, then write `check_stability.sh` (executable, `chmod +x`) that runs the binary 5 times, sha256sums the output each time, and exits 1 on any mismatch.
