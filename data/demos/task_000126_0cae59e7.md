Treat this as two independent data sources that must be reconciled, not merged with a naive join. In Go: (1) load and parse graph.json, build two lookup structures — a map from User -> set of Roles (from HAS_ROLE edges) and a map from Role -> set of Resources (from CAN_ACCESS edges); then compute the *true* access set as all (user, resource) pairs reachable by composing those two maps (a two-hop join, not a direct edge lookup — there is no direct User->Resource edge in the graph). (2) open audit.db with the sqlite3 driver and SELECT all rows from reported_access. (3) for each reported (user_id, resource_id) pair, flag it as a violation only if it is NOT present in the true access set computed in step 1 — this catches pairs the flawed reporting tool invented via an implicit cross join. Sort the resulting violation list by user_id then resource_id before marshaling to JSON with the exact field names `user_id` and `resource_id`. Remember to `go mod init` and `go get github.com/mattn/go-sqlite3` (which needs cgo enabled) before building, and use struct tags so json.Marshal emits lowercase keys matching the required schema.
