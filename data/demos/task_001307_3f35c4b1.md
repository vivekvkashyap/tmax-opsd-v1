Because the panic is intermittent, a bare `git bisect run go run main.go input.csv` will unreliably mark bad commits as good. Write a wrapper shell script (e.g. `/home/user/check.sh`) that loops running the program against `input.csv` many times (20-30 iterations), checking the exit code each iteration; if any iteration returns non-zero (panic), the script exits 1 (bad), otherwise it exits 0 (good) after all iterations pass. Make the script executable, then run `git bisect start`, mark current `HEAD` as bad and the initial commit as good, and run `git bisect run ./check.sh`. Let bisect walk the full commit range automatically rather than testing commits by hand — with dozens of commits, manual bisection is error-prone and slow. Once bisect reports the first bad commit, capture its full 40-character SHA with `git rev-parse <commit>` (not the abbreviated form shown in bisect's output), run `git bisect reset` to restore the working tree, and write only that hash (no trailing newline) to `/home/user/bad_commit.txt`.
