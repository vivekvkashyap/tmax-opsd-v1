Treat this as three separable subsystems and verify each independently before chaining them. First, transcribe /app/manifest_audio.wav yourself (e.g. build/run whisper.cpp with a small model, or another local STT tool) and read off the spoken 3-digit IDs from the transcript text — don't guess or hardcode IDs. Second, write the C++ curator to, per approved ID: open /app/repo/artifact_<ID>.tar.gz, validate it's a well-formed gzip/tar stream (e.g. attempt full decompression/extraction and check for errors rather than just checking the header) before trusting it, log corrupted ones with the ID to /home/user/corruption.log and continue rather than aborting, and for valid ones extract raw_binary.bin to a `.tmp` file in /home/user/staging/ then use `rename()`/`std::filesystem::rename` (an atomic same-filesystem move) to name it release_<ID>_stable.bin — this atomicity requirement is the part naive solutions skip. Finally, for the compression step, note that plain `tar -czf` will not hit the small-size target on highly-compressible dummy data — use `tar -cJf` (xz) or `tar` piped through `xz -9` (or `xz -9e`) to get aggressive compression, and confirm the extracted archive contains exactly the expected stable.bin files before finishing.
