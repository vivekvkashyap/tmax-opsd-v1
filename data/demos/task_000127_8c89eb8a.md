This is a classic bucketing/sharding exercise — build the index once, then make queries touch only one shard. In optimize.sh: `rm -rf`/recreate /home/user/index, then stream sales.csv (skip the header) and for each row extract the customer_id's last two characters (e.g. with `${customer_id: -2}` in bash or `cut`/`sed`) and append the full CSV row to `/home/user/index/part_<those-two-digits>.csv` — do this with a single awk pass over the file for speed rather than 100 separate greps, e.g. `awk -F, 'NR>1{suffix=substr($2,length($2)-1,2); print > ("/home/user/index/part_"suffix".csv")}' sales.csv`, remembering awk needs `close()` or enough open file handles. In fast_query.sh: take $1, compute the same 2-digit suffix to pick exactly one part_XX.csv file, then run a single awk/grep+awk filter on just that file for `$2==customer_id && $4=="Electronics"` summing column 5, printing the total (handle the zero-match case still producing 0.00, matching the expected decimal format). The key mistake to avoid is having fast_query.sh scan the whole sales.csv or all 100 partitions — verification greps the script for `part_` and checks wall-clock-style narrow I/O.
