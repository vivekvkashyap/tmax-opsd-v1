Work this as two separate deliverables. First, forensics: use `ffmpeg -i /app/incident.mp4 -vf fps=1 /tmp/frames/frame_%03d.png` (or a higher fps if text is only visible briefly) to dump frames, then inspect them (OCR with `tesseract` or visually) to read the exact command line the attacker ran — note which tool, which flag carries the destination, and what URI scheme is used for that destination. Second, build the detector generically, not from the one example you saw: parse the command-line string into whitespace-separated tokens, locate the destination-carrying flag (both its long and short forms) and grab its argument, then check whether that argument begins with a disallowed network scheme (e.g. `tcp://` or `udp://`) versus a local path or an authorized scheme (e.g. `ssh://`, plain filesystem paths). Reject (exit 1) only on the disallowed-scheme match; accept (exit 0) everything else, including other command names, missing flags, or malformed input. Read the file path from argv[1], read its full contents safely (bounded buffer, check fopen/read return values), and avoid crashing on short/garbage input — the detector will be run against many clean and malicious samples, not just the one you observed.
