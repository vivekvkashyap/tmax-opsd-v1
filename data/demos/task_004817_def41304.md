This has three largely independent deliverables — build and smoke-test each before combining. (1) SSH: generate a user-writable sshd config (custom `Port 2222`, `ListenAddress 127.0.0.1`, `PasswordAuthentication no`, `PubkeyAuthentication yes`, and exactly the specified `KexAlgorithms`/`Ciphers`/`MACs` lines), generate host keys with `ssh-keygen` into a location your config points at, append the given public key line to the `user` account's `~/.ssh/authorized_keys` (create the dir with 700 perms, file with 600), then launch `/usr/sbin/sshd -f <your_config> -D &` (or without -D, backgrounded) as your own user — this requires no root since it's a non-privileged port and a custom config, not the system sshd. Verify the algorithm restrictions actually work by test-connecting with `ssh -o KexAlgorithms=... -p 2222` for both the allowed algorithm (should succeed) and a disallowed one (should be rejected) before moving on. (2) Run `nmap -p 2222 --script ssh2-enum-algos -oX /home/user/ssh_scan.xml 127.0.0.1`. (3) For the audit binary, use `strings /app/legacy_audit_sender` and/or `objdump -d` to spot base64-like tables or XOR-loop patterns in the disassembly, or simplest: run it against a listener (e.g. `nc -l 8080` or a minimal logging HTTP server) and capture the exact POST body it sends for a known/controllable input, then reverse the transform by trial-decoding (try base64-decode then single-byte XOR against every key 0-255, checking which key yields printable ASCII) — don't guess the key, derive it from a captured sample. Once you know the transform, write your web service to reverse it (base64-decode, then XOR each byte) and append the decoded plaintext + newline to the log file. Leave both services running in the background.
