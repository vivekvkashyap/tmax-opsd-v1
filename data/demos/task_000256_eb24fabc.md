Work the three subsystems independently, testing each in isolation before chaining them. For SSH: since StrictModes cannot be disabled, remember sshd silently rejects authorized_keys (and its parent .ssh and home directory) if permissions or ACLs are too permissive — check `ls -la` and `getfacl` on `~/.ssh` and `~/.ssh/authorized_keys`, strip any group/other write bits and any extra ACL entries (`setfacl -b`), and tighten to owner-only (e.g. 700/600). Check sshd's own error log (run sshd in foreground with `-d` or inspect syslog) rather than guessing. For the router, inspect `/home/user/app/router/routes.conf`'s existing entries to learn its syntax, then correct the `backend` alias to point at the actual backend port rather than assuming the file format. Verify each piece with a direct test (raw `curl` to the backend, then through the router, then via an SSH `-L` tunnel) before combining them. For the health-check script, use `date` for the timestamp, `curl -s -o /dev/null -w '%{http_code}'` for the status code, a `while true; sleep 5` loop, and background it with `nohup ... &` (or `disown`) so it survives your session and keeps appending to the log file.
