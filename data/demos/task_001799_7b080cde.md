Build this as an ETL-then-serve pipeline in Rust, testing each phase before moving to the next. First run `/app/start_services.sh` and confirm both Redis (6379) and the file server (8080) are actually reachable before writing any Rust. Scaffold with `cargo new`, add `reqwest` (blocking or async), `encoding_rs` for legacy decoding, `redis`, `axum`/`tokio` for the server. For each CSV, fetch the raw bytes (not `.text()`, which assumes UTF-8) and decode explicitly with `encoding_rs::SHIFT_JIS` or `encoding_rs::WINDOWS_1252`/ISO-8859-1 equivalent before parsing `KEY,VALUE` lines — decoding with the wrong charset or letting reqwest auto-assume UTF-8 will corrupt the non-ASCII text. Store each pair in Redis under `locale:{lang}:{key}`, and after each file finishes, append the exact log line with the correct count to `/home/user/pipeline.log` (open in append mode). Only after ETL fully completes should you start the axum server on 127.0.0.1:9000 with a `/translate/:lang/:key` handler that does a Redis `GET` on `locale:{lang}:{key}` and returns 200 with the value as plain UTF-8 text, or 404 if the key is missing. Launch the compiled binary in the background (e.g. `nohup ... &`) so it keeps listening after your shell command returns.
