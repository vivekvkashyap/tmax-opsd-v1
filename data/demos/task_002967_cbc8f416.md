The naive trap here is processing the CSV line-by-line with grep/sed, which silently mangles or drops the multi-line quoted `translation` field — you must first collapse each logical CSV record onto one physical line before doing any field-level work. Do this with an `awk` pass that tracks quote parity: read the file character-by-character or line-by-line while counting unescaped double-quotes seen so far; while the running count is odd, you're inside a quoted field, so buffer the line, replace the literal newline you're about to add with the two-character sequence `\n`, and keep reading until the quote count becomes even again, then emit the joined record. Once every record is exactly one line, use `awk -F,` (or a small state machine, since a naive split on `,` in the reassembled line is now safe as this dataset only quotes for embedded newlines) to split into the 4 fields, strip surrounding quotes, mask the email's local-part with `sed -E 's/^(.)[^@]*/\1***/'` on the fourth field, escape any `'` in the translation field by doubling it, and emit the `INSERT INTO locales (...) VALUES (...)` line per record, skipping the header. Wire it all into the single script file, chmod +x it, and run it to produce the SQL file.
