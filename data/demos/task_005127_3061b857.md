This is a robustness/adversarial-input problem, not a feature-completeness one — the sanitiser will be run against many malformed and many clean dependency files, and it must classify every single one correctly, so favor conservative, well-tested checks over clever parsing. Structure the C++ program as: (1) read the file line by line, split each non-empty line on whitespace into target + dependency tokens; (2) for every token, validate UTF-8 by walking the byte sequence and checking continuation-byte rules (reject on any decode failure or embedded `\0`), and separately reject any token containing the literal substrings `../` or `..\` via simple substring search; (3) build a directed graph (target -> each dependency) and detect cycles with a standard DFS using three-color (white/gray/black) marking rather than a naive "no visited set" recursion — since the legacy binary's own bug is exactly missing cycle/visited tracking, don't repeat that mistake. Fail closed: any parse ambiguity, cycle, or invalid token should produce REJECT and exit 1; only a fully clean, acyclic graph should print ACCEPT and exit 0. Write the Makefile with `g++ -O2 -std=c++17 sanitise_deps.cpp -o sanitise_deps`, then test against both a clean and an obviously cyclic/traversal input by hand before trusting it against the full corpus.
