The critical trap is that hard links share the same inode, so any in-place edit (e.g. reopening a renamed file with `fopen(path, "r+")` and overwriting bytes) will also mutate the backup copy's content since it's literally the same data on disk. To avoid this, in `doc_manager.c`: (1) first create `/home/user/backup` and, for every `doc_XX.md`, call `link(src, backup_dst)` to snapshot it — this must happen before any renaming or editing. (2) Then `rename()` each `doc_XX.md` to `rel_XX.md` in `/home/user/docs` — renaming doesn't affect the hard-linked backup's inode/content. (3) For the content edit, do NOT edit the renamed file in place; instead read its full content into memory, perform the string replacements (`State: WORK_IN_PROGRESS` -> `State: RELEASED`, and rewrite the `Location:` line to point at the new `rel_XX.md` path), write the modified content to a fresh temporary file, then `rename()` that temp file over `rel_XX.md`. This breaks the inode link between the renamed docs file and the backup, so the backup remains byte-for-byte original while `rel_XX.md` gets new content and a new inode. Verify by comparing `stat -c %i` of a backup file vs its corresponding renamed file — they must differ.
