Treat this as a systematic multi-bug debugging exercise: read `run.sh` and `main.go` fully before touching anything, and fix each bug independently, verifying with `go run` after each change rather than guessing all fixes at once. (1) Compare the env var name `run.sh` exports against the one `main.go` reads with `os.Getenv` — a mismatched name means the config path is always empty, so align them exactly. (2) The base64 decoding function must match the encoding scheme actually used to produce the WAL's `url` field — inspect a sample base64 string's characters; if it contains only standard base64 alphabet characters (no `-`/`_`), the URL-safe decoder is the wrong choice and you need the standard one. (3) For the "up" counting loop, check the loop bound against slice length carefully — an off-by-one that includes an out-of-range index will panic; use `<` not `<=`, or safer, use `range`. (4) For the corrupted trailing WAL line, change the unmarshal-error path from `panic` to a `continue` so parsing keeps going and simply skips that record. After fixing, run `bash run.sh` (or the equivalent) and inspect the generated `uptime_report.json` to confirm it reflects only the valid, decodable entries.
