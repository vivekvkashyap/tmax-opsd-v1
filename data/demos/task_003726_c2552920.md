Build the pipeline in stages and test each independently before wiring them together. First use `tshark -r /app/traffic.pcap -Y http -T fields -e http.file_data` (or reassemble via tcp stream export) to dump each HTTP request body/payload to its own file in a temp dir, deduping identical content. Then write `/home/user/scan_waf.sh` to loop (or `xargs -P N`) over payload files, invoking the untrusted binary strictly inside `bwrap --unshare-net --unshare-pid --ro-bind /usr /usr --ro-bind /lib /lib --ro-bind /lib64 /lib64 --ro-bind /etc/ld.so.cache /etc/ld.so.cache --ro-bind <payload> <payload> --tmpfs /tmp -- /app/waf_evaluator <payload>` — do not grant write or home access beyond the one input file. Since the binary may behave differently with network access, always run with `--unshare-net` to force the deterministic offline path and avoid hangs. Detect a crash by checking the exit status: a SIGSEGV shows up as exit code 139 (or via `$?` >128 indicating a fatal signal). For every crashing payload, compute `md5sum` of the payload file and append the hash to `/home/user/crashing_hashes.txt`. Validate against F1 by first testing on a handful of payloads you construct yourself with headers/lengths resembling the described trigger pattern, not by guessing the exact vulnerable condition.
