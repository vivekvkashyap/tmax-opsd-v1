Reverse-engineer the oracle before writing any implementation. Generate many 12,288-byte RGB24 files with different, deliberately simple patterns (all-zero, all-max, single-channel-high, gradients) and run `/app/oracle_analyzer` on each, recording which channel average and threshold triggers PASS/FAIL/FLAKY/CORRUPT and their priority order when multiple thresholds could apply — binary-search the threshold value with files that isolate one channel at a time rather than guessing. Once you have a hypothesis, write `libframeparser.c` implementing `decode_ci_status` with that exact channel-average/threshold/priority logic, compile it as a shared object (`gcc -shared -fPIC -o libframeparser.so ...`), and write `analyzer_cli.c` that reads exactly 12,288 bytes, calls into the library (link with `-L. -lframeparser` and set `rpath` or `LD_LIBRARY_PATH` so it resolves at runtime), and prints the status with a trailing newline. Before finishing, fuzz your own binary against the oracle with random byte content across many trials to catch edge cases (e.g. channel averages exactly at a boundary, integer truncation vs rounding) — the real grader does this at large scale. Finally, use `ffmpeg` to extract raw RGB24 frames from `/app/ci_test_run.mp4` and run each through your CLI to build `ci_report.txt`.
